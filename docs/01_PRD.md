## 1. 产品定位

把 `FindJobs-Agent` 的 AI 引擎从"应聘者被评估"翻转为"招聘者做评估"。HR 批量导入简历，系统自动解析、打标、画像、为 JD 荐人、AI 预筛面试；管理者用 BI 看板监控团队效能。

---

## 2. 角色与权限

| 角色 | 能做什么 | 数据可见范围 |
|---|---|---|
| HR 专员 | 传简历、看候选人画像、推进流水线、发起 AI 预筛 | 仅自己负责的候选人/岗位 |
| 招聘主管/HRD | 分配岗位、看 BI 看板、考核效能 | 全团队 |
| 用人部门面试官 | 看推荐简历、填面试评价 | 被指派的候选人 |
| 系统管理员 | 账号/权限、标签体系、模型配置 | 系统级 |

## 3. 功能模块与对应代码

### 模块 A：批量简历导入与处理 ★必备1

| 子功能 | 复用/新增 | 落到代码 |
|---|---|---|
| A1 批量上传（多文件/zip） | 扩展 | 改造 `POST /api/resume/upload` 收多文件；新增任务队列 |
| A2 解析结构化 | 复用 | `resume_parser.py` 包装 `parse_batch()`，输出统一 JSON |
| A3 打标评分 | 复用 | `tag_rate.py`，标签写 `candidate_tags`，兼容 `Python , 5 , AI` 格式 |
| A4 候选人画像 | 新增 | 聚合结构化字段+标签→候选人判断卡片；默认展示推荐判断、核心亮点、核心技能和待确认风险，雷达降级为辅助，全量技能标签折叠查看 |
| A5 来源渠道标准化 | 新增 | 上传批次记录候选人来源，支持 BOSS直聘、58同城、猎聘、鱼泡直聘等选项，供候选人库筛选和 BI 渠道质量统计；来源信息默认选填收起 |

**用户故事**：作为 HR 专员，我拖拽上传一个含 80 份简历的 zip，系统先把解析成功的候选人沉淀到简历库。等我确认岗位后，再到简历库选择目标岗位查看适配候选人，并把合适的人加入岗位流程。

**验收**：单批 ≥100 份；关键字段抽取准确率 ≥90%；失败项可单独重试；PII 字段脱敏存储；上传页不要求选择岗位，上传成功后统一进入简历库；简历库顶部入口使用“岗位匹配”，筛选区使用“目标岗位”和“入流程状态”，避免和普通筛选重复；候选人来源、内推人/猎头联系人和批次备注为选填信息，默认收起；BI 中同一渠道不会因简称或别名被拆分。

### 模块 B：岗位管理与人岗匹配 ★必备1

| 子功能 | 复用/新增 | 落到代码 |
|---|---|---|
| B1 JD 录入与结构化 | 复用 | `job_agent.py` 抽取学历/专业/技能要求 |
| B2 岗找人匹配（反转） | 反转 | 输入 JD 画像，遍历简历池算匹配度，输出排序候选人+理由 |
| B3 智能荐人 | 新增 | 新 JD 发布自动从存量简历池召回（激活沉睡简历） |
| B4 岗位缺口引导与恢复 | 新增 | 关键流程无可选岗位时引导新建岗位；已关闭岗位可查看并恢复在招 |

**用户故事**：作为 HR，我录入一条"高级后端"JD，系统结构化出要求后，从现有简历池返回按匹配度排序的候选人列表，每个标注"为什么匹配/缺什么"。

**验收**：给定 JD 返回 Top-N 候选人 + 可解释匹配理由；岗位匹配页提供“AI 推荐 / 全部候选人”视角，HR 可按姓名、公司、岗位、学校、技能或邮箱搜索全库候选人，并按匹配度、入流程状态、匹配技能和缺失技能筛选；AI 没推荐到的人也能人工补找、查看当前岗位匹配预览并加入流程；用人需求、简历上传、候选人流程、面试安排在没有可选岗位时有明确新建岗位入口；误关闭岗位后可在已关闭岗位列表恢复；下拉框有数据但找不到目标项时，也能看到新建岗位、上传简历或联系管理员的入口。

### 模块 C：AI 预筛面试（反转）★必备1

复用 `AI_interviewer.py`，把"帮候选人练面试给建议"反转为"对候选人作答评分"。

| 接口 | 原用途 | 反转后 |
|---|---|---|
| `/api/interview/start` | 候选人开始练习 | HR 对某候选人发起预筛，按 JD 出题 |
| `/api/interview/answer` | 候选人提交答案得建议 | 候选人异步作答，AI 评分 |

**输出**：预筛报告 = 分项评分 + 亮点 + 疑点（识别"包装简历"）+ 是否建议进真人面试。

**验收**：候选人完成异步作答后，HR 收到结构化报告。

### 模块 D：招聘流水线 ★新增

Kanban：`待初筛 → AI预筛 → 部门面试 → Offer → 入职 / 淘汰`。每次流转写 `events` 表（供 BI）。

**用户故事**：作为 HR，我把通过 AI 预筛的候选人卡片拖到"部门面试"，系统记录流转时间与操作人。

**误操作补救**：候选人推进错阶段或误淘汰时，HR/经理/管理员在管道里使用“修正阶段”，必须填写修正原因。系统追加一条带“阶段修正”标记的流水，当前阶段和 BI 当前存量按最新阶段计算，历史记录不删除。

**面试反馈归口**：面试阶段的人工反馈统一在“面试任务”页填写，但该页面不作为招聘专员、经理或管理员的左侧一级入口。候选人流程和工作台待办保留跳转入口并携带岗位/候选人参数，面试任务页自动定位到待反馈项；面试官在“我的面试”任务卡点击“填写反馈”后，页面自动切到待处理并滚动到对应反馈表。AI 预筛只作为辅助参考，不作为人工安排和反馈的主入口。

**流程直达**：招聘专员工作台“今日待办”可直接进入候选人流程的业务待反馈、面试中、Offer 等阶段；候选人流程默认展示有候选人且更需要处理的阶段，空阶段必须提供去非空阶段、去匹配更多候选人或上传简历的下一步入口。

**需求补救**：用人需求误关闭、误暂停或业务重新确认继续招聘时，HR/经理/管理员可恢复需求，系统同步把关联岗位恢复为在招；需求优先级可在 A/B/C 之间重新调整，并记录调整原因，避免误降级后只能重新建需求。关闭、恢复和调整优先级必须通过站内确认弹窗填写业务原因，并说明这次操作会影响活跃需求、岗位状态或后续复盘。

### 模块 E：BI 监控看板 ★必备2（难点，详见 [03](./03_BI看板设计.md)）

管理者看团队当前阶段分布、各专员效能对比、渠道质量、瓶颈环节和数据质量提醒，可三层下钻。周期指标按“本期入库简历 cohort”统计，当前阶段分布按全量最新阶段统计；当前流程人数不含已入职/已淘汰，入职占比用全流程去重人数做分母。

**验收**：主管/管理员可看全局 BI；招聘专员只能看自己负责范围，协作岗位只返回自己负责候选人的岗位漏斗；面试官不开放岗位级或专员级 BI。专员绩效统一按 `candidates.owner_hr_id` 归属，团队均值采用总入职/总简历的加权口径，来源渠道按标准渠道聚合，Offer 数高于面试进入数等异常要在总览和专员下钻显示数据质量提醒。

BI 页面必须用白话解释协同归属：HR 负责候选人推进，推进人只看操作留痕，面试官负责反馈闭环，用人部门负责岗位协同；异常数据旁给出下一步动作建议。

**AI 助手示例**：AI 助手首页示例问题按角色展示。招聘专员看到自己负责候选人的卡点和待反馈问题，经理/负责人看到团队漏斗和专员推进问题，管理员看到审计、权限和 AI 边界问题，避免用户被引导到无权限入口。

### 模块 F：系统支撑 ★新增

- F1 RBAC（角色权限隔离）
- F2 操作埋点（写 `events`，BI 数据源）
- F3 PII 脱敏 + 访问审计 + 候选人授权告知
- F4 候选人负责人转派：经理/管理员通过招聘专员下拉框选择新负责人，填写转派原因，不让业务用户输入用户 ID。

### 模块 G：BOSS 直聘招聘闭环 ★新增

依赖外部 CLI `boss`（boss-cli，从 GitHub 源码安装，必须配置；详见 `RUNNING.md`/`DEPLOYMENT.md`）。`/boss` 页面在原有「搜候选人/推荐候选人/岗位管理」基础上新增「收件箱·闭环」工作台，把外部招聘动作沉淀进本系统流水线。

| 子功能 | 复用/新增 | 落到代码 |
|---|---|---|
| G1 收件箱拉取 | 新增 | `boss recruiter inbox` → `/api/boss/inbox`，按 `label`（沟通中/新招呼/全部）+`limit` 拉取已沟通候选人 |
| G2 批量导入候选人库 | 新增 | `BossPipelineService.batch_import`（`/api/boss/candidates/batch-import`）：勾选后按 geek_id 去重、限量+间隔下载完整简历原文，落 `Candidate.resume_json`（原文 + 解析基础字段），可绑定目标岗位入 `pending` 阶段；命中限流即停并回报 `stopped_reason` |
| G3 AI 简历初筛 | 新增 | `PreScreenService.evaluate_resume` + `BossPipelineService.ai_screen`（`/api/boss/candidates/ai-screen`）：按岗位 JD 对简历 LLM 评估出分/建议，写 `Interview(ai_report.type=resume_screen)`，推进到 `ai_screen` 阶段 |
| G4 面试邀请（人工确认+双写） | 新增 | `BossPipelineService.invite_interview`（`/api/boss/candidates/invite-interview`）：HR 手动确认后先调 `boss recruiter invite-interview`，成功才写系统 `InterviewAssignment` 并推进 `interview` 阶段；BOSS 失败不落系统脏数据 |

**用户故事**：作为 HR 专员，我在 `/boss`「收件箱·闭环」里勾选几位已沟通的候选人一键导入候选人库，对照目标岗位让 AI 给出初筛分数和建议，再对通过的人逐个确认后发面试邀请——邀请同时同步到 BOSS 和本系统流水线，后续面试/反馈/Offer 沿用现有流程。

**验收**：导入按 geek_id 去重、限量+间隔执行、限流即停并回报原因；AI 初筛分数与建议落库且候选人进入 `ai_screen`；面试邀请需人工确认，BOSS 成功才双写、失败不落系统记录；面试官无权调用闭环写操作（403）；boss-cli 未安装时整组接口返回 503 并在前端引导安装。

### 模块 H：AI 助手高可用增强 ★新增

解决原 AI 助手「切换会话数据丢失、无历史记录、无调用日志」三大痛点，把 AI 助手做成可审计、会话不丢的高可用模块。详细计划见 `docs/ai_assistant_upgrade_plan.md`。

| 子功能 | 落到代码 | 说明 |
|---|---|---|
| H1 会话管理 | `POST/GET/PATCH/DELETE /api/agent/conversations` | 新建/列表(分页+归档过滤)/详情/重命名/软删归档；按 user_id 隔离，跨用户 403 |
| H2 历史会话侧边栏 | `AgentPage.tsx` 左侧栏 + `agentChat.tsx` Context | 会话列表、切换、新建、删除、重命名；切换有内存缓存避免闪烁；刷新不丢(后端为唯一事实源) |
| H3 AI 调用日志 | `AgentCallLog` 表 + `LLMClient` 埋点 + chat/tool_write 写入 | 记录每次 AI 调用的输入/输出/工具链/模型/token/耗时/状态；log 写入失败不阻断对话 |
| H4 调用日志审计面板 | `GET /api/agent/call-logs[/<id>]` + `/admin/agent-logs` 页 | 管理员看全部、非管理员只看自己；统计卡+筛选+分页+展开详情 |
| H5 多轮上下文完整 | `AgentPage.buildHistory` | 回传上一轮 tool_calls/thoughts 摘要(控 token)，避免重复调工具/上下文断裂 |

**用户故事**：作为 HR 专员，我在 AI 助手左侧能看到所有历史会话，点击切换不会丢失当前对话，刷新页面历史还在；作为管理员，我能在「AI 调用日志」页看到每次 AI 调用的输入输出、用了多少 token、是否出错、调了哪些工具，便于审计和排障。

**验收**：切换会话后消息正确恢复、刷新不丢；AI 调用后 `agent_call_logs` 表有记录(chat + tool_write 两种 kind)；管理员可见全部调用日志、非管理员只看自己；log 写入失败不影响对话返回。

## 5. 端到端验收场景

1. HR 上传 100 份简历 → 异步完成解析+打标+画像。
2. 录入 JD → 返回排序候选人列表 + 匹配理由。
3. 候选人完成 AI 预筛 → HR 收到评估报告。
4. 候选人在 Kanban 推进 → 全程留痕。
5. 管理者在 BI 看板看各专员漏斗与效能并下钻。
6. 专员只能看自己候选人；PII 脱敏、审计日志可查。
