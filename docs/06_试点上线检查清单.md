# 06 · 试点上线检查清单（v0.1-internal → 公司服务器小范围试点）

> 版本：2026-06-17 ｜ 状态：待执行
> 目标：把当前 `v0.1-internal` 内测基线，安全地部署到公司服务器，供少数 HR 用**真实数据**小范围试点。
> 范围：**只做配置加固 + 部署**。本阶段**不接 OA**（OA 自动承接需求见单独的对接问题清单），需求阶段先由 HR 手工录入。

## 2. 逐项展开

### C1 · 生产密钥与环境变量 〔配置加固〕

**为什么**：现在 `backend/app/config.py` 里 `JWT_SECRET` 默认值是 `dev-secret-change-in-prod`、`SECRET_KEY` 默认 `dev-secret`。沿用默认值任何人都能伪造登录令牌，等于无认证。

**动作**：
1. 在服务器生成强随机密钥（不要写进代码、不要进 git）：
   ```bash
   python3 -c "import secrets; print(secrets.token_urlsafe(48))"
   ```
2. 写入 `backend/.env`（参考 DEPLOYMENT.md §3.3 的 `.env` 用法）：
   ```env
   FLASK_DEBUG=false
   JWT_SECRET=<上一步生成的强随机串>
   JWT_EXPIRY_HOURS=8
   ```
3. 确认 `.env` 已在 `.gitignore` 中（项目已忽略，勿提交真实密钥）。

**验收**：`grep JWT_SECRET backend/.env` 不再是 `change-in-prod`；用旧 demo token 访问 `/api/jobs` 返回 401。

**红线核对**：只改环境变量取值，不改 `config.py` 读取逻辑、不改认证机制 → 符合 R5。

---

### C2 · CORS 收紧 〔配置加固〕

**为什么**：`backend/app/__init__.py` 现在是 `CORS(app)`，等于允许任意网站跨域调用接口。生产必须限制到公司前端域名。

**动作**：把 `CORS(app)` 改为按环境变量限定来源（不改架构，仅收紧参数）：
```python
CORS(app, origins=os.environ.get("CORS_ORIGINS", "").split(","))
```
并在 `.env` 配置：
```env
CORS_ORIGINS=https://zhipin.内网域名
```
> 若采用 R2 的「Flask 同源托管 dist」方案，前后端同源，可进一步收紧或免跨域。

**验收**：从非白名单域名发起的浏览器请求被 CORS 拦截；同源页面正常。

**红线核对**：仍用 flask-cors，不换中间件、不改路由 → 符合 R3 / R5。

---

### C3 · 数据库切 PostgreSQL + 定时备份 〔部署〕

**为什么**：开发默认 SQLite（单文件、并发弱、易随容器重置丢失）。多人试点用真实简历，必须切到 DEPLOYMENT.md §6 已认可的 PostgreSQL，并做备份。

**动作**：
1. 在公司服务器建 PostgreSQL 库与账号。
2. `.env` 配置（DEPLOYMENT.md §6 方案 A 写法）：
   ```env
   DATABASE_URL=postgresql://user:pass@host:5432/zhipin
   ```
   > 代码会自动把 `postgresql://` 转成 SQLAlchemy 的 `postgresql+psycopg://` 驱动形式。
3. 首次建表 + 不要带 demo 数据（建表逻辑沿用现有 `create_app`，见 C6）。
4. 配置每日备份：`pg_dump` 定时任务，保留至少 7 天。

**验收**：后端连上 PostgreSQL 启动无报错；`pg_dump` 能导出；删库后能从备份恢复。

**红线核对**：仅切换 SQLAlchemy 的引擎 URL，schema 不变、ORM 不变 → 符合 R4。

---

### C4 · HTTPS 〔部署〕

**为什么**：简历、候选人手机/邮箱属敏感个人信息，裸 HTTP 传输有泄露风险。公司内网最低也要 HTTPS。

**动作**（二选一，均不改应用代码）：
- **方案 A（推荐）**：Nginx 反向代理在前，挂公司证书，转发到 `127.0.0.1:5000`。
- **方案 B**：若已有公司统一网关/负载均衡终结 TLS，则后端只监听内网回环。

> Cloudflare 免费隧道（DEPLOYMENT.md §7）仅适合临时演示，链接会变，**不用于正式试点**。

**验收**：浏览器访问为 `https://`，证书有效；HTTP 自动跳转 HTTPS。

**红线核对**：TLS 在应用之外终结，部署仍是「一个 Flask + 一个前端」 → 符合 R2。

仓库已提供 Nginx 样例：

```bash
deploy/nginx/zhipin.conf.example
```

使用前替换域名和证书路径，再把后端转发到 `127.0.0.1:5000`。

### C5 · LLM Key 配置 + 数据外发合规确认 〔合规〕

**为什么**：简历解析、JD 结构化、AI 面试、AI 助手都会把内容发给外部 LLM（DeepSeek/OpenAI）。真实简历含个人信息，**外发前必须经公司确认合规**——这是上线最关键的非技术卡点。

**动作**：
1. **先确认合规**：向公司法务/IT 确认「能否把候选人简历内容发送给外部 LLM API」。若不允许，本次试点可先只用离线功能（候选人库/岗位/流程/BI 离线可用，见 DEPLOYMENT.md §5 表），AI 解析等暂缓。
2. 合规通过后，在 `.env` 配置生产 Key（沿用现有字段，不改调用代码）：
   ```env
   LLM_PROVIDER=deepseek
   LLM_API_URL=https://api.deepseek.com/v1/chat/completions
   OPENAI_API_KEY=<生产 key>
   ```
3. 设月度预算/调用上限，避免试点期间费用失控。

**验收**：合规结论有书面记录；上传一份真实简历能正常解析（或确认已走离线模式）。

**红线核对**：沿用既有 DeepSeek/LangGraph 路线与 `.env` 字段，不换模型框架 → 符合 R3。

### C6 · 清演示数据、删 demo 账号、建真账号 〔数据〕

**为什么**：种子数据全是演示候选人，demo 账号密码都是公开的 `demo1234`，必须清除后再放真实数据。

**动作**：
1. 生产库**不要**运行 `seed_dev.py`（那是演示数据）。
2. 删除所有 `@demo.com` 账号。
3. 由 admin 为试点 HR 建真实账号，强制设强密码（沿用现有 auth 改密流程）。
4. 真实岗位/需求由试点 HR 在系统内**手工录入**（本阶段不接 OA）。

**验收**：用户列表无 demo 账号；候选人库为空或仅含真实数据；`demo1234` 无法登录。

**红线核对**：仅数据层操作，不改 schema、不改 auth 逻辑 → 符合 R4 / R5。

### C7 · 自动化门禁全绿 + 真人冒烟 〔验收〕

**为什么**：上线前必须证明代码是绿的、流程是通的。沿用 05 收口计划的固定门禁，不新增工具（R6）。

**动作 1 — 自动化门禁**（05 收口计划 §4 原命令，在你本机/CI 跑）：
```bash
./.venv312/bin/python -m pytest backend/tests base_agent/tests -q
cd frontend && npm run lint
cd frontend && npm run typecheck
cd frontend && npm run build
for f in frontend/tests/*.test.mjs; do node "$f" || exit $?; done
python3 -m pip_audit -r backend/requirements.txt
cd frontend && npm audit --audit-level=moderate
```
> 允许存在的已知 warning 见 05 收口计划 §4（SQLAlchemy legacy、utcnow 弃用等），其余须为绿。

**动作 2 — 真人冒烟**：在部署好的服务器上，按 05 收口计划 §5 的 10 步真人点一遍（管理员建账号 → HR 传真简历 → 候选人库筛选 → 编辑刷新匹配 → 建岗匹配批量入流程 → Pipeline 推进 → 面试官反馈 → 管理者看 BI → AI 助手查询 → 越权路径验证）。

**验收**：门禁命令全绿（除允许的 warning）；10 步冒烟全部通过，越权路径确认被拦截。

**红线核对**：原样沿用既有门禁与冒烟脚本 → 符合 R6。

### C8 · 部署执行 〔部署〕

按 DEPLOYMENT.md §6 既定方案，不引入新拓扑：
```bash
cd frontend && npm run build          # 生成 dist，由 Flask 同源托管
cd backend && gunicorn -w 2 -b 127.0.0.1:5000 --timeout 120 --keep-alive 5 "run:app"
```
配 systemd 开机自启（DEPLOYMENT.md §6 方案 C），前面挂 C4 的 HTTPS 反代。

**验收**：服务器重启后服务自动拉起；试点 HR 能通过 HTTPS 域名访问完整系统。

### C9 · 上线后观察项 〔运维〕

- 日志：确认 `logs/` 有应用日志，定期查错误。
- 备份：确认 C3 的每日备份真在跑，且能恢复。数据库和 `uploads/` 要一起备份。
- 回滚：保留上一可用版本（git tag `v0.1-internal` 已在），出问题能快速切回。
- 反馈：建一个试点 HR 反馈收集入口（群/文档），为 v0.2 与 OA 对接攒需求。

备份脚本：

```bash
cd backend
python scripts/backup_pilot_data.py --dry-run
python scripts/backup_pilot_data.py
```

建议用 cron 每天跑一次，`BACKUP_DIR` 指到服务器专门的备份目录。

### C10 · 安全响应头 + 热点接口限流 〔部署加固〕

**为什么**：登录、AI 助手、简历上传最容易被误刷或恶意刷。安全响应头可以让浏览器多一层保护。

**动作**：

1. 应用侧已增加兜底安全头和轻量限流，生产 `.env` 保持：
   ```env
   SECURITY_HEADERS_ENABLED=true
   RATE_LIMIT_ENABLED=true
   RATE_LIMIT_LOGIN=10
   RATE_LIMIT_AGENT_CHAT=20
   RATE_LIMIT_RESUME_UPLOAD=8
   ```
2. Nginx 层使用 `deploy/nginx/zhipin.conf.example`，重点保护：
   - `/api/auth/login`
   - `/api/agent/chat`
   - `/api/resume/upload`

**验收**：连续错误登录超过阈值返回 429；浏览器响应头里能看到 `X-Frame-Options`、`X-Content-Type-Options`、`Content-Security-Policy`。
