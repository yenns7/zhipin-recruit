> 版本：2026-06-17 ｜ 状态：待执行

## 2. 逐项展开

### C1 · 生产密钥与环境变量 〔配置加固〕

**为什么**：现在 `backend/app/config.py` 里 `JWT_SECRET` 默认值是 `dev-secret-change-in-prod`、`SECRET_KEY` 默认 `dev-secret`。沿用默认值任何人都能伪造登录令牌，等于无认证。

**动作**：
1. 在服务器生成强随机密钥（不要写进代码、不要进 git）：
   ```bash
   python3 -c "import secrets; print(secrets.token_urlsafe(48))"
   ```
2. 写入 `backend/.env`（参考 DEPLOYMENT.md §3.3 的 `.env` 用法）：
   ```env
   FLASK_DEBUG=false
   JWT_SECRET=<上一步生成的强随机串>
   JWT_EXPIRY_HOURS=8
   # BOSS 账号 Cookie 加密密钥（云部署必填，固定值；留空则每次重启生成临时密钥，已存 BOSS 账号失效）
   # 生成：python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
   FIELD_ENCRYPTION_KEY=<生成的固定 Fernet 密钥>
   ```
3. 确认 `.env` 已在 `.gitignore` 中（项目已忽略，勿提交真实密钥）。

**验收**：`grep JWT_SECRET backend/.env` 不再是 `change-in-prod`；用旧 demo token 访问 `/api/jobs` 返回 401；`FIELD_ENCRYPTION_KEY` 已设固定值，重启后端后已导入的 BOSS 账号仍能正常调用（不报解密失败）。

**红线核对**：只改环境变量取值，不改 `config.py` 读取逻辑、不改认证机制 → 符合 R5。

### C2 · CORS 收紧 〔配置加固〕

**为什么**：`backend/app/__init__.py` 现在是 `CORS(app)`，等于允许任意网站跨域调用接口。生产必须限制到公司前端域名。

**动作**：把 `CORS(app)` 改为按环境变量限定来源（不改架构，仅收紧参数）：
```python
CORS(app, origins=os.environ.get("CORS_ORIGINS", "").split(","))
```
并在 `.env` 配置：
```env
CORS_ORIGINS=https://zhipin.内网域名
```
> 若采用 R2 的「Flask 同源托管 dist」方案，前后端同源，可进一步收紧或免跨域。

**验收**：从非白名单域名发起的浏览器请求被 CORS 拦截；同源页面正常。

**红线核对**：仍用 flask-cors，不换中间件、不改路由 → 符合 R3 / R5。

### C3 · 数据库切 PostgreSQL + 定时备份 〔部署〕

**为什么**：开发默认 SQLite（单文件、并发弱、易随容器重置丢失）。多人试点用真实简历，必须切到 DEPLOYMENT.md §6 已认可的 PostgreSQL，并做备份。

**动作**：
1. 在公司服务器建 PostgreSQL 库与账号。
2. `.env` 配置（DEPLOYMENT.md §6 方案 A 写法）：
   ```env
   DATABASE_URL=postgresql://user:pass@host:5432/zhipin
   ```
   > 代码会自动把 `postgresql://` 转成 SQLAlchemy 的 `postgresql+psycopg://` 驱动形式。
3. 首次建表 + 不要带 demo 数据（建表逻辑沿用现有 `create_app`，见 C6）。
4. 配置每日备份：`pg_dump` 定时任务，保留至少 7 天。

**验收**：后端连上 PostgreSQL 启动无报错；`pg_dump` 能导出；删库后能从备份恢复。

**红线核对**：仅切换 SQLAlchemy 的引擎 URL，schema 不变、ORM 不变 → 符合 R4。

### C4 · HTTPS 〔部署〕

**为什么**：简历、候选人手机/邮箱属敏感个人信息，裸 HTTP 传输有泄露风险。公司内网最低也要 HTTPS。

**动作**（二选一，均不改应用代码）：
- **方案 A（推荐）**：Nginx 反向代理在前，挂公司证书，转发到 `127.0.0.1:5000`。
- **方案 B**：若已有公司统一网关/负载均衡终结 TLS，则后端只监听内网回环。

> Cloudflare 免费隧道（DEPLOYMENT.md §7）仅适合临时演示，链接会变，**不用于正式试点**。

**验收**：浏览器访问为 `https://`，证书有效；HTTP 自动跳转 HTTPS。

**红线核对**：TLS 在应用之外终结，部署仍是「一个 Flask + 一个前端」 → 符合 R2。

仓库已提供 Nginx 样例：

```bash
deploy/nginx/zhipin.conf.example
```

使用前替换域名和证书路径，再把后端转发到 `127.0.0.1:5000`。

### C5 · LLM Key 配置 + 数据外发合规确认 〔合规〕

**为什么**：简历解析、JD 结构化、AI 面试、AI 助手都会把内容发给外部 LLM（DeepSeek/OpenAI）。真实简历含个人信息，**外发前必须经公司确认合规**——这是上线最关键的非技术卡点。

**动作**：
1. **先确认合规**：向公司法务/IT 确认「能否把候选人简历内容发送给外部 LLM API」。若不允许，本次试点可先只用离线功能（候选人库/岗位/流程/BI 离线可用，见 DEPLOYMENT.md §5 表），AI 解析等暂缓。
2. 合规通过后，在 `.env` 配置生产 Key（沿用现有字段，不改调用代码）：
   ```env
   LLM_PROVIDER=deepseek
   LLM_API_URL=https://api.deepseek.com/v1/chat/completions
   OPENAI_API_KEY=<生产 key>
   ```
3. 设月度预算/调用上限，避免试点期间费用失控。

**验收**：合规结论有书面记录；上传一份真实简历能正常解析（或确认已走离线模式）。

**红线核对**：沿用既有 DeepSeek/LangGraph 路线与 `.env` 字段，不换模型框架 → 符合 R3。

### C6 · 清演示数据、删 demo 账号、建真账号 〔数据〕

**为什么**：种子数据用于 MVP 内部试用；正式试点前应清理样例候选人，并用管理员创建真实试点账号。

**动作**：
1. 生产库**不要**运行 `seed_dev.py`（那是演示数据）。
2. 先预览清理范围：
   ```bash
   python backend/scripts/cleanup_demo_data.py --dry-run
   ```
3. 确认备份目录、数据库和上传目录无误后，再由负责人确认执行：
   ```bash
   python backend/scripts/cleanup_demo_data.py --confirm
   ```
   脚本会先调用备份逻辑，再删除 `@mvp.local` demo 账号及其关联业务数据，并清空 `backend/uploads/`、`uploads/` 文件；schema 保留。
4. 删除历史演示账号或临时试用账号。
5. 由 admin 在系统设置/用户管理中创建试点账号；登录页不开放自注册，只提示使用管理员分配账号。
6. 招聘专员尽量一人一个账号，BI 看板按用户 ID 统计个人推进量和结果指标。
7. 面试官账号只用于查看“我的面试”和填写反馈，不作为 HR 管理账号使用；不开放全量简历库、候选人流程和 AI 助手主入口，也不能直接推进 Offer 或淘汰。
8. 如试点用户忘记密码，由 admin 在用户管理中重置密码，再通过企业微信/线下安全方式发给本人。
9. 真实岗位/需求由试点 HR 在系统内**手工录入**（本阶段不接 OA）。

**验收**：用户列表无历史演示账号；候选人库为空或仅含真实数据；旧演示密码无法登录；登录页无自注册入口；admin 可创建账号并重置密码；招聘专员登录后工作台可看到“我的本月业绩”和“今日待办”，点击待办可直达候选人流程对应阶段或面试待反馈列表；面试官登录后左侧只看到工作台和“我的面试”主入口，任务卡可直接点击“填写反馈”，填写反馈时不出现“推进 Offer/淘汰”按钮。

**红线核对**：仅数据层操作，不改 schema、不改 auth 逻辑 → 符合 R4 / R5。

### C7 · 自动化门禁全绿 + 真人冒烟 〔验收〕

**为什么**：上线前必须证明代码是绿的、流程是通的。沿用 05 收口计划的固定门禁，不新增工具（R6）。

**动作 1 — 自动化门禁**（05 收口计划 §4 原命令，在你本机/CI 跑）：
```bash
python backend/scripts/check_pilot_readiness.py
./.venv312/bin/python -m pytest backend/tests base_agent/tests -q
cd frontend && npm run lint
cd frontend && npm run typecheck
cd frontend && npm run build
for f in frontend/tests/*.test.mjs; do node "$f" || exit $?; done
python3 -m pip_audit -r backend/requirements.txt
cd frontend && npm audit --audit-level=moderate
```
> 自检脚本只读检查生产 `.env` 硬门槛，不打印密钥或数据库密码。允许存在的已知 warning 见 05 收口计划 §4（SQLAlchemy legacy、utcnow 弃用等），其余须为绿。

**动作 2 — 真人冒烟**：在部署好的服务器上，按 05 收口计划 §5 的 10 步真人点一遍（管理员建账号 → HR 上传简历 → 候选人库筛选 → 编辑刷新匹配 → 建岗匹配批量入流程 → Pipeline 推进并确认默认打开非空阶段 → 面试官登录只看“我的面试”并从任务卡直接填写反馈且不能推进 Offer/淘汰 → 招聘专员看个人工作台并从“今日待办”直达对应阶段 → 管理者看 BI 首屏数字、周期筛选、数据质量提醒和专员绩效，确认右上角不再出现“怎么看数据” → AI 助手按角色展示示例问题并可查询 → 越权路径验证）。

**验收**：门禁命令全绿（除允许的 warning）；10 步冒烟全部通过，越权路径确认被拦截。

**AI 助手高可用增强验收**（模块 H）：
- AI 助手左侧「会话历史」可见，新建/切换/重命名/删除均可，切换后消息正确恢复、刷新不丢。
- 在 AI 助手发起一次对话后，管理员在「AI 调用日志」页（`/admin/agent-logs`）能看到该次调用记录（含模型/token/耗时/状态/工具链），点击可展开完整输入输出；非管理员只能看自己的记录。
- AI 提议写操作并人工确认执行后，`agent_call_logs` 表有一条 `kind=tool_write` 记录，且 `conversation_id` 非空（关联到所在会话）。

**红线核对**：原样沿用既有门禁与冒烟脚本 → 符合 R6。

### C10 · 安全响应头 + 热点接口限流 〔部署加固〕

**为什么**：登录、AI 助手、简历上传最容易被误刷或恶意刷。安全响应头可以让浏览器多一层保护。

**动作**：

1. 应用侧已增加兜底安全头和轻量限流，生产 `.env` 保持：
   ```env
   SECURITY_HEADERS_ENABLED=true
   RATE_LIMIT_ENABLED=true
   RATE_LIMIT_LOGIN=10
   RATE_LIMIT_AGENT_CHAT=20
   RATE_LIMIT_RESUME_UPLOAD=8
   ```
2. Nginx 层使用 `deploy/nginx/zhipin.conf.example`，重点保护：
   - `/api/auth/login`
   - `/api/agent/chat`
   - `/api/resume/upload`

**验收**：连续错误登录超过阈值返回 429；浏览器响应头里能看到 `X-Frame-Options`、`X-Content-Type-Options`、`Content-Security-Policy`。

### C11 · 用户可用性缺口检查 〔产品验收〕

**为什么**：自动化测试通过，只能说明主流程大体可运行；真实 HR 试用时更容易卡在“岗位没得选、候选人为空、面试官账号没建、岗位被关闭后找不到、推进错了不知道怎么改”这些边界场景。试点前必须把这些卡点提前点一遍，不能只按理想数据判断。

**动作**：

1. 用招聘专员账号检查：
   - 新建用人需求时，如果没有目标岗位，是否有清楚提示和“去新建岗位”的入口。
   - 工作台“今日待办”是否能直接进入业务待反馈、面试中、Offer 或待补反馈，而不是只进入普通列表首页。
   - 上传简历时是否只进入简历库；如果要推进某个岗位，是否能知道下一步去简历库筛选后加入岗位流程。
   - 岗位匹配页是否能在 AI 推荐之外切到“全部候选人”，按已知姓名、公司、学校、岗位、技能或邮箱人工补找，并只把当前筛选后勾选的人批量加入流程。
   - 候选人加入岗位流程时，如果候选人为空或岗位为空，是否有下一步入口；列表不为空但找不到目标项时，也要有“上传简历 / 新建岗位”的入口。
   - 候选人流程默认是否避开空阶段；用户手动点到空阶段时，页面是否提示去非空阶段、去匹配更多候选人或上传简历。
   - 候选人推进错阶段或淘汰错人时，是否能用“修正阶段”补救，并强制填写修正原因；时间线是否显示“阶段修正”。
2. 用招聘经理/负责人账号检查：
   - BI 看板是否能解释“谁推荐了多少人、多少进入面试、多少面试通过、多少 Offer/入职、哪里卡住”，并能用“协同归属”说明 HR 推进、面试反馈闭环、用人部门协同和推进操作留痕分别看什么。
   - 岗位被关闭后是否还能找到历史数据，是否需要恢复岗位。
   - 如果多个招聘专员协作同一岗位，数据归属是否说得清。
   - 候选人负责人是否能通过招聘专员下拉框转派，并要求填写转派原因，不要求输入用户 ID。
3. 用面试官账号检查：
   - 左侧菜单是否只保留面试官真正需要的入口。
   - 如果没有分配面试，空页面是否说明原因。
   - 有待反馈任务时，任务卡是否直接提供“填写反馈”，点击后是否定位到候选人和岗位对应的反馈表。
   - 填写反馈后，HR/经理是否能看到反馈沉淀。
4. 用管理员账号检查：
   - 系统设置是否按“账号管理 / 审计日志 / AI 边界”分区；账号管理是否先展示成员列表，需要时再展开创建账号表单。
   - 能否创建账号、停用账号、重置密码。
   - 如果没有可选面试官，管理员是否能看到用户管理入口；非管理员是否能看到“联系管理员创建或启用面试官账号”。
   - 面试官账号停用后，是否不能继续被新安排面试；旧页面或旧请求是否被后端拦截。
   - 用户管理为空时是否提示先创建管理员、招聘专员和面试官，并提醒试点建议一人一个账号。
   - 当前是否需要“查看已关闭岗位 / 恢复岗位 / 限定试点岗位范围”的管理能力；暂不做的要列为已知限制。

**验收**：四类账号各走一遍后，形成“可直接发给试点用户的问题说明”：哪些能用、哪些是已知限制、用户卡住时应该找谁。

### C12 · 逆向验收十角度检查 〔产品验收〕

**为什么**：正常成功路径只能证明“会用的人能走通”。试点更容易暴露的是点错、没数据、权限不一致、数字解释不清、部署版本不一致。每次说“全面测试”时，都按以下十个角度反向验收。

**动作**：

1. 反向路径：点错、选错、刷新、半路退出、数据为空、接口返回错误时，页面是否告诉用户下一步。
2. 不同角色：招聘专员、主管、面试官、管理员分别看得到什么、能改什么、不能越权做什么；AI 助手首页示例问题也要符合角色权限，不能把招聘专员引导到经理/管理员才有权限的团队 BI 或治理入口。
3. 数据闭环：上传简历、加入岗位、推进、面试、Offer、入职后，候选人详情、管道、面试任务页、BI 是否一致。
4. 脏数据：重复候选人、解析失败、来源别名、停用账号、关闭岗位、缺手机号/邮箱时是否有兜底。
5. BI 反查：每个数字能否追到候选人、岗位、负责人、推进人、面试官和时间范围。
6. 误操作恢复：关闭、淘汰、推进、转派、停用账号、重置密码后，是否有补救或明确风险说明。
7. 下拉框目标缺失：完全没数据、有数据但没有目标、目标被关闭/停用三种情况都要测。
8. 重复点击/并发：同一按钮连点、重复加入流程、重复提交反馈、两个人同时操作时是否产生重复或错乱。
9. 部署版本一致：本地代码、commit、tag、部署包、文档必须是同一版，避免本机有但服务器没有。
10. 非技术用户理解：页面不能要求用户理解技术字段；错误提示必须是业务语言，并说明找谁或去哪一步处理。

**验收**：输出 P0/P1/P2 清单。P0 是会阻断试点或造成权限/数据归属错误的问题；P1 是用户会卡住或主管解释不清的问题；P2 是体验优化或试点中可继续观察的问题。

### C13 · BOSS 账号登录改为浏览器导入 〔产品验收〕

**为什么**：系统部署在云服务器，后端读不到使用者本机浏览器的登录态。而 BOSS 招聘端鉴权需要 `__zp_stoken__` + `wt2` + `wbg` + `zp_at` 一组完整 Cookie，其中 `__zp_stoken__` 由页面 JS 生成、会话 Cookie 多为 HttpOnly。bookmarklet 拿不到 HttpOnly Cookie，**只有浏览器扩展（`chrome.cookies`）能采集完整 Cookie**。扫码登录现在通过 Camoufox 无头浏览器在服务端补全 `__zp_stoken__`，但 Camoufox 受 BOSS 反爬影响并非 100% 成功，故「从浏览器导入账号」仍是最稳路径，扫码作为补充入口。

**动作**：

1. 部署/分发 `browser-extension/`（智聘 · BOSS Cookie 采集器）给试点招聘人员，按 README 在 Chrome/Edge 以开发者模式加载。
2. 试点人员在本机浏览器登录 BOSS 招聘端 → 扩展「一键采集并复制」→ 平台「从浏览器导入账号」粘贴保存。
3. 确认 `FIELD_ENCRYPTION_KEY` 已固定（见 C1），否则重启后导入的账号会失效。
4. 若启用扫码登录的 Camoufox 补全：安装 `kabi-boss-cli[browser]` + `python -m camoufox fetch`，确认 `BOSS_QR_STOKEN_HYDRATE` 未被设为 `0`。

**验收**：缺 `__zp_stoken__` 的 Cookie 导入被拒（提示改用扩展，不落库）；完整 Cookie 导入后账号激活、`/api/boss/status` 正常；后端重启后已导入账号仍可用；扫码登录状态轮询经历 `pending → scanned → stoken（补全中）→ done`，补全成功则保存含 stoken 的完整账号，Camoufox 补不出 stoken 时 confirm 返回 409 `needs_stoken` 并引导改用浏览器导入、不保存残缺凭证；试点说明里写清扩展安装步骤、Cookie 用途与留存期限。

**红线核对**：新增浏览器导入接口与扩展，不改既有认证机制与数据归属逻辑；登录方式变更已同步 `RUNNING.md` / `DEPLOYMENT.md`。
