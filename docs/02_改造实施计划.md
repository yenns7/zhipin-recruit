## 1. 逐文件改造清单（核心）

按"动多少"分四类，每条给出**改造动作**和**为什么**。

### 1.1 直接复用（包一层即可）

**`tag_rate.py` — 技能打标评分**
- 动作：保留打分逻辑；新增写库，标签存入 `candidate_tags(candidate_id, tag, score)`。
- 兼容：原已支持 `Python , 5 , AI`、`Python %> 5 , AI`、`Python: 5` 三种格式，直接沿用。
- 为什么：评分逻辑与应聘/招聘视角无关，纯复用。

**`tech_taxonomy.json` / `all_labels.csv` — 分类与标签库**
- 动作：原样导入作为系统初始标签体系。
- 为什么：技能分类是中立资产。

### 1.2 反转改造（逻辑反向）

**匹配逻辑（原"人找岗" → "岗找人"）**
- 现状：原系统给候选人推荐岗位。
- 动作：输入一条 JD 画像，遍历 `candidates` 简历池逐一算匹配度，输出**按分排序的候选人列表 + 匹配理由**。
- 实现建议：先用标签集合重合度 + 技能评分加权做召回打底，再用 LLM 对 Top-N 重排并生成"为什么匹配/缺什么"。
- 为什么：这是"应聘者→招聘者"翻转最核心的一处。

**`AI_interviewer.py` — 面试官反转**
- 现状：帮候选人练面试、给改进建议（教学口吻）。
- 动作：改 prompt 角色为"考官"，对候选人作答**评分**而非建议；产出预筛报告（分项评分/亮点/疑点/是否建议进真人面试）。复用其"按 JD 生成题目""智能追问"能力识别包装简历。
- 为什么：引擎一样，主客体对调。

### 1.3 扩展改造（加能力）

**`resume_parser.py` — 批量+异步**
- 动作：新增 `parse_batch(files[])`；接 zip 解压；解析结果统一 JSON 入 `candidates`；失败项标记可重试。
- 为什么：原是单文件，HR 场景要批量百份。

**`job_agent.py` — 转为 JD 管理后端**
- 动作：复用其抽取学历/专业/技能的能力，服务于 B1 岗位录入（不再服务于爬来的岗位）。

**`api_server.py`(Flask) — 分层+鉴权+埋点**
- 动作：拆 `api/service/repository` 三层；加 JWT 鉴权中间件、埋点中间件（每个写操作落 `events`）、分页。
- 现有端点演进：
  | 原端点 | 改造 |
  |---|---|
  | `POST /api/resume/upload` | 收多文件/zip，入队异步 |
  | `POST /api/resume/analyze` | 触发批量打标+画像 |
  | `GET /api/jobs` | 改为企业内部 JD 列表 |
  | `POST /api/interview/start` | HR 对候选人发起预筛 |
  | `POST /api/interview/answer` | 候选人异步作答→AI 评分 |
  | 新增 `GET /api/candidates` | 候选人列表（带权限过滤） |
  | 新增 `POST /api/match` | 岗找人匹配 |
  | 新增 `GET /api/bi/*` | BI 看板数据 |

**`FrontEnd/`(React+TS) — 页面演进**
- `JobsPage` → 候选人流水线 Kanban；`ResumePage` → 批量上传+画像；`InterviewPage` → 预筛报告查看；新增 `BIDashboard`、`JobManage`。

### 1.4 弃用

**`job_crawler_v2.py` / `_selenium.py` / `pipeline.py` 爬取段**
- 动作：本期不进主流程（假设 A1：简历靠导入，不爬外部岗位）。代码保留备用。
- 为什么：方向相反——原是爬公开岗位给个人，企业是收候选人简历。

## 3. 目标架构

```
[React 前端]
  ├ 批量上传/画像页  ├ 流水线 Kanban  ├ 预筛报告  ├ BI 看板  ├ JD 管理
        │
[Flask API 网关]  ← JWT 鉴权中间件 + 埋点中间件（写 events）
  ├ 简历服务 ─→ [任务队列 Celery/Redis] ─→ worker: resume_parser→tag_rate→画像
  ├ 岗位服务（job_agent 结构化 JD）
  ├ 匹配服务（反转：JD→简历池→排序）
  ├ 预筛服务（AI_interviewer 反转评估）
  ├ 流水线服务（状态机，写 events）
  └ BI 服务（events 聚合→指标）
        │
[PostgreSQL] + [Redis] + [对象存储:原始简历文件] + [LLM 网关:可换模型]
```

> 相比原仓库新增：任务队列、鉴权、埋点、关系库、对象存储。原 Flask 单文件升为分层服务。

## 4. 数据模型（核心表）

| 表 | 关键字段 | 来源 |
|---|---|---|
| `users` | id, name, role | 新增 |

| `events` | actor_id, action, entity_id, ts | 埋点（BI 源） |
| `audit_logs` | actor_id, target, action, ts | 合规 |

## 5. 分阶段实施（4–6 人团队，工期为相对估算待核实）

**阶段 0 地基（约 1 周）**
迁入可复用文件并补单测；搭 Docker+PostgreSQL+Redis；抽 `llm_client` 为可换模型层。
交付：骨架跑通，`resume_parser`/`tag_rate` 自测过。

**阶段 1 简历闭环 MVP（约 2-3 周）★必备1**
A1-A4（批量上传/解析/打标/画像）+ B1-B2（JD 管理/岗找人匹配）。
交付：传简历→看画像→给 JD 看排序候选人。

**阶段 2 AI 预筛+流水线（约 2 周）★必备1闭环**
C（`AI_interviewer` 反转）+ D（Kanban）+ F2（埋点）。
交付：候选人作答→AI 报告→Kanban 推进，全程留痕。

**阶段 3 BI 看板（约 2-3 周）★必备2 难点**
E1-E3（漏斗/专员效能/瓶颈）+ F1（RBAC）。详见 [03](./03_BI看板设计.md)。
交付：管理者看板可下钻到专员/岗位。

**阶段 4 合规加固（约 1 周）**
F3（PII 脱敏/审计）+ 权限渗透测试 + 100 份/批压测。
交付：可私有化交付 1.0。

## 6. 风险与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| 简历 PII 合规 | 高 | 脱敏+授权+审计+私有化部署 |
| 原仓库无测试 | 高 | 复用前补单测，service 层隔离 |
| 反转匹配效果不达预期 | 中 | 规则召回兜底 + LLM 重排迭代 |
| BI 埋点口径不一 | 中 | 阶段2 先定指标字典再做看板 |
| AI 解析/评分准确率 | 中 | 人工复核入口 + 评测集回归 |
| LLM 成本/限流 | 中 | 异步批处理+缓存+可换国产模型 |
