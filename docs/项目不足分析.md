## 概述

本报告对智聘招聘系统进行了全面代码审查，覆盖后端 API 层、数据模型层、中间件、前端页面组件、状态管理、测试体系和构建部署配置。共发现 **39 项不足**，按严重程度分为 4 级。其中 6 项为阻塞上线的关键缺陷，8 项为严重影响可维护性的高优问题。

## CRITICAL — 阻塞上线/高风险

### C1. 无数据库迁移框架

- **位置**：`backend/app/__init__.py:41`
- **现状**：使用 `db.create_all()` 初始化表结构，配合 3 个手写 `_ensure_*_columns()` 函数执行原生 `ALTER TABLE` SQL
- **影响**：生产环境 schema 变更不可逆、不可控，多人协作时容易冲突
- **建议**：引入 Alembic，建立 migration 基线，后续所有 schema 变更走 migration 流程

### C2. 无健康检查端点

- **位置**：全局（无 `/health` 或 `/healthz` 端点）
- **现状**：没有任何探活接口
- **影响**：Kubernetes/ECS/负载均衡无法检测服务存活状态，故障时无法自动恢复
- **建议**：添加 `GET /health` 端点，返回数据库连接状态、关键依赖可达性

### C3. 无 Dockerfile / docker-compose

- **位置**：项目根目录（`base_agent/` 下有 Dockerfile，但主项目无）
- **现状**：无容器化配置
- **影响**：部署依赖本地环境，不可复现，无法标准化交付
- **建议**：添加 `Dockerfile` + `docker-compose.yml`，定义 backend、frontend、数据库服务

### C4. 前端无真正可运行的测试

- **位置**：`frontend/tests/`（50 个 `.mjs` 文件）
- **现状**：所有测试均为 `fs.readFileSync` + 正则断言，检查源码字符串是否存在。无组件渲染、无逻辑测试、无测试框架
- **影响**：零回归保护，代码改动全靠人肉验证
- **建议**：引入 Vitest + @testing-library/react，至少覆盖核心页面关键路径

### C5. 前端无 ErrorBoundary

- **位置**：`frontend/src/App.tsx`
- **现状**：10 个路由全部使用 `React.lazy()` + `Suspense`，但无 `ErrorBoundary` 兜底
- **影响**：chunk 加载失败（网络异常、构建问题）时整个应用白屏，不可恢复
- **建议**：在 `Suspense` 外层包裹 `ErrorBoundary`，提供重试按钮和降级 UI

### C6. JWT 客户端裸存 localStorage

- **位置**：`frontend/src/lib/auth.tsx:15-66`
- **现状**：token 存 localStorage，无刷新机制、无过期校验、角色通过 `as Role` 强转无验证
- **影响**：token 过期无感知，角色可被伪造（编辑 localStorage）
- **建议**：添加 token 过期检查、refresh token 机制、服务端 session 校验

## HIGH — 严重影响质量和可维护性

### H1. `sys.path.insert` 散布 6 个文件

- **位置**：`config.py:6`、`jobs.py:11`、`agent_service.py:24`、`interview_service.py:4`、`match_service.py:4`、`resume_service.py:5`
- **现状**：6 个文件重复执行 `sys.path.insert(0, str(BASE_AGENT_DIR))` 将 base_agent 加入搜索路径
- **影响**：import 顺序不可控，测试环境脆弱，打包困难
- **建议**：将 base_agent 配置为可编辑包（`pip install -e ./base_agent`），消除运行时路径注入

### H2. 内存限流器在多 worker 下失效

- **位置**：`backend/app/middleware/rate_limit.py:8`
- **现状**：使用 `defaultdict(deque)` 做进程内限流，gunicorn `-w 4` 时每个 worker 独立计数
- **影响**：实际限流阈值被放大 4 倍，生产环境限流形同虚设
- **建议**：改用 Redis-backed 限流（如 flask-limiter + Redis backend）

### H3. DB commit 无异常处理

- **位置**：全局 API 层（50+ 处 `db.session.commit()`）
- **现状**：绝大多数 commit 调用无 try/except 包裹
- **影响**：约束冲突、连接断开时直接抛 500，可能泄露敏感 traceback
- **建议**：在全局错误处理器中捕获 `SQLAlchemyError`，或在关键路径添加 try/except + 回滚

### H4. 异常静默吞掉 (17 处)

- **位置**：
  - `backend/app/api/auth.py:23` — 密码校验失败静默返回 False
  - `backend/app/api/jobs.py:57` — LLM 提取失败静默返回 `{}`
  - `backend/app/services/interview_service.py:57,67,77` — LLM 解析失败返回默认值
  - `backend/app/services/boss_service.py:217,573` — subprocess/解密错误吞掉
- **影响**：问题排查无据，静默失败导致脏数据
- **建议**：所有 `except Exception:` 必须至少记录 `logger.warning` 或 `logger.error`

### H5. 前端 API 返回 `Promise<unknown>`

- **位置**：`frontend/src/lib/api.ts:604-648`（7 个 BOSS 端点）
- **现状**：BOSS 相关 API 函数返回类型为 `unknown`，`api<T>` 函数用 `data as T` 强转无运行时校验
- **影响**：TypeScript 类型系统形同虚设，运行时数据结构不匹配时静默产生 undefined
- **建议**：为 BOSS 端点定义响应类型，引入 zod 或 io-ts 做运行时校验

### H6. 巨型页面组件 (600-1100 行)

- **位置**：
  - `frontend/src/pages/BiPage.tsx` — 1100 行，8+ 内联子组件
  - `frontend/src/pages/AgentPage.tsx` — 907 行，6 内联子组件
  - `frontend/src/pages/JobsPage.tsx` — 847 行
  - `frontend/src/features/candidates/pages/CandidatesPage.tsx` — 793 行
  - `frontend/src/pages/DashboardPage.tsx` — 695 行
- **影响**：不可测试、不可复用、git 冲突频繁
- **建议**：拆分为独立组件文件，按功能域组织到 `components/` 或 `features/` 下

### H7. 代码重复 (3 处)

- **KpiCard 双实现**：`BiPage.tsx` 内联实现 vs `DashboardPage.tsx:279-312` 不同 API 的 KpiCard
- **城市选项常量 3 份**：`JobsPage.tsx` (`COMMON_JOB_CITY_OPTIONS`)、`CandidatesPage.tsx` (`COMMON_CITY_OPTIONS`)、`BossPage.tsx` 各维护一份
- **错误渲染未统一**：有 `ErrorState` 组件 (`src/components/ui/ErrorState.tsx`) 但 BiPage 等仍手写错误 div
- **影响**：修 bug 要改多处，UX 不一致
- **建议**：提取共享组件和常量到 `components/ui/` 和 `lib/constants/`

### H8. `useAsync` 缺少核心能力

- **位置**：`frontend/src/lib/useAsync.ts`
- **现状**：无请求缓存、无自动重试、无智能取消、无去重。`DashboardPage.tsx:208-275` 另起炉灶自定义 `useDashboardStats`
- **影响**：同一数据多次 fetch，无容错，每个页面独立实现数据获取
- **建议**：引入 SWR 或 TanStack Query 替代自研 useAsync

## MEDIUM — 技术债积累

### M1. models.py 单文件 365 行

- **位置**：`backend/app/models.py`
- **现状**：20+ 个数据库模型（User、Candidate、Job、PipelineStage、BossAccount、Conversation、AgentCallLog、TalentMap 等）全部定义在一个文件
- **建议**：按域拆分到 `models/user.py`、`models/candidate.py`、`models/boss.py`、`models/agent.py`

### M2. bi.py 1100 行上帝文件

- **位置**：`backend/app/api/bi.py`
- **现状**：包含 8+ 个独立指标计算函数（漏斗、员工绩效、渠道质量、需求健康度、简历消耗、面试问责、管理告警、数据质量警告）
- **建议**：拆分为 `services/bi/` 下独立指标模块，API 层只做路由和参数解析

### M3. 无输入验证框架

- **位置**：全局 API 层
- **现状**：无 marshmallow、pydantic 或 cerberus。所有验证靠 `data.get()` + 手工类型检查
- **建议**：引入 marshmallow 或 pydantic 做请求/响应 schema 验证

### M4. 日志配置不一致

- **位置**：全局
- **现状**：30 个 Python 文件中仅 7 个使用 logging（agent.py、agent_service.py、boss_pipeline_service.py、boss_service.py、boss_qr_service.py、crypto.py、resume.py）
- **建议**：在 `__init__.py` 中配置 `dictConfig`，所有模块统一使用 `logging.getLogger(__name__)`

### M5. 无 HTTP 请求/响应日志中间件

- **位置**：`backend/app/middleware/`
- **现状**：仅有 auth、rate_limit、events 三个中间件，无 HTTP 访问日志
- **建议**：添加 before_request/after_request 日志，记录方法、路径、状态码、耗时

### M6. 密码哈希代码重复

- **位置**：`backend/app/api/auth.py:15-27` + `backend/app/api/admin.py:16-17`
- **现状**：auth.py 定义 `_hash`/`_verify`，admin.py 另有 `_hash_password`，auth.py 还保留 SHA-256 旧版兼容
- **建议**：统一到 `services/crypto.py` 或 `utils/auth.py`，移除旧版 SHA-256 兼容

### M7. JWT_SECRET 默认值不一致

- **位置**：`backend/app/config.py:29` vs `config.py:77`
- **现状**：`JWT_SECRET` 默认 `"dev-secret-change-in-prod"`，`SECRET_KEY` 默认 `"dev-secret"`，两个变量指向不同默认值
- **建议**：统一为同一个环境变量，生产环境强制校验

### M8. CSP 允许 unsafe-inline + unsafe-eval

- **位置**：`backend/app/__init__.py:63-64`
- **现状**：`script-src 'self' 'unsafe-inline' 'unsafe-eval'` 基本架空了 CSP
- **建议**：移除 `unsafe-eval`，对 inline script 改用 nonce 或 hash

### M9. Agent 单例非线程安全

- **位置**：`backend/app/api/agent.py:18-36`
- **现状**：`_get_agent()` 用全局变量 + None 检查实现懒初始化，无锁
- **影响**：gunicorn 多线程 worker 下可能重复初始化
- **建议**：用 `threading.Lock` 保护，或在 app startup 时一次性初始化

### M10. match_service 全量加载候选人到内存

- **位置**：`backend/app/services/match_service.py:28`
- **现状**：`candidates = (candidate_query or Candidate.query).all()` 加载全部候选人
- **建议**：改为数据库端匹配或分批加载

### M11. list_jobs / list_interviews 无分页

- **位置**：`backend/app/api/jobs.py:183`、`backend/app/api/interview.py:420-441`
- **现状**：返回所有记录，在 Python 端排序
- **建议**：添加 page/page_size 参数，数据库端分页 + 排序

### M12. N+1 查询

- **位置**：`backend/app/api/interview.py:412-418`（cname/jtitle/uname 循环内 get）、`backend/app/api/candidates.py:187`（循环内 get UploadBatch/Job）
- **建议**：使用 `joinedload` 或 `selectinload` 预加载关联对象

### M13. 仅 agent_call_logs 有索引

- **位置**：`backend/app/models.py:306-309`
- **现状**：只有 `agent_call_logs` 定义了 3 个索引。`candidates.owner_hr_id`、`pipeline_stages.candidate_id/job_id`、`events.ts`、`interview_feedback.candidate_id/job_id` 等高频查询列无索引
- **建议**：为高频过滤/ JOIN 列添加索引

### M14. Celery 配置了但无实际 task

- **位置**：`backend/app/config.py:64-66`、`backend/app/workers/`（空 `__init__.py`）
- **现状**：配置了 `memory://` broker + `CELERY_TASK_ALWAYS_EAGER=True`，但无 task 定义
- **建议**：要么移除 Celery 依赖，要么实际实现异步 task（如简历解析、LLM 调用）

### M15. 前端 `defaultRouteForRole` 始终返回 `/`

- **位置**：`frontend/src/lib/nav.ts`
- **现状**：角色默认路由函数无实际逻辑，所有角色登录后都跳 `/`
- **建议**：按角色返回合理默认页（如招聘专员 → 候选人列表，管理员 → 仪表盘）

### M16. 类型定义 `| string` 扩散

- **位置**：`frontend/src/types/index.ts:221`（JobListItem.status）、`:502`（OfferRecord.approval_status）、`:666`（BiManagerAlert.kind）
- **现状**：枚举/联合类型被 `| string` 放宽，失去类型保护
- **建议**：移除 `| string`，用枚举或字面量联合类型严格约束

### M17. 无 React.memo / 无列表虚拟化

- **位置**：全局前端
- **现状**：无 `React.memo` 使用，长列表（候选人表格、流水线卡片）无虚拟化
- **建议**：对纯展示组件添加 memo，长列表引入 react-window 或 react-virtuoso

### M18. GSAP 用于简单动画

- **位置**：`frontend/src/lib/motion.ts`、`AppShell.tsx`
- **现状**：GSAP (3.15.0) 用于侧边栏动画和页面过渡
- **建议**：简单动画改用 CSS transitions 或 framer-motion（更轻量且与 React 集成更好）
