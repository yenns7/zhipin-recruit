# v0.1-internal 版本收口计划

> 目标：把当前项目从“持续堆功能”收束成一个可演示、可内测、可继续迭代的稳定基线。
> 版本定位：内部演示 / 内测版，不定义为生产上线版。

## 1. 当前阶段判断

当前项目已经具备完整 MVP 闭环：

- 登录与角色权限
- 简历上传、解析、候选人库
- 岗位管理、人岗匹配
- 招聘流程 Pipeline
- AI 面试与面试官反馈
- BI 看板与管理提醒
- AI 助手读写工具
- 需求管理与演示数据整理

当前重点不是继续新增功能，而是：

- 收口权限和上传安全边界
- 把混在一起的改动按业务包分开
- 固定一套自动化验证命令
- 打一个内部稳定版本点

## 2. 建议版本名

建议打版名：

```text
v0.1-internal
```

含义：

- `0.1`：第一个可完整跑通的招聘 MVP 版本
- `internal`：内部演示和内测可用
- 不代表生产级合规、审计、对象存储、多租户和高并发能力已经完成

## 3. 交付包拆分

### 包 A：安全与权限收口

目的：避免不同角色通过 API 或 AI 助手绕过页面权限。

包含：

- JWT 每次请求按数据库刷新用户状态和角色
- 候选人可见性统一到 access helper
- 匹配、Pipeline、面试、Offer、AI 助手读写工具加权限边界
- 上传、重解析、旧 `/api/match` 入口、需求管理接口加权限限制
- `cryptography` 升级到 `48.0.1`

核心文件：

- `backend/app/api/access.py`
- `backend/app/middleware/auth.py`
- `backend/app/api/jobs.py`
- `backend/app/api/match.py`
- `backend/app/api/pipeline.py`
- `backend/app/api/interview.py`
- `backend/app/api/resume.py`
- `backend/app/api/demands.py`
- `backend/app/services/agent_service.py`
- `backend/requirements.txt`
- `backend/tests/test_access_control_hardening.py`
- `backend/tests/test_auth_security.py`
- `backend/tests/test_security_hardening_next.py`

### 包 B：简历上传与候选人画像

目的：让 HR 能修正解析失败或不完整的候选人资料，上传入口更稳。

包含：

- 上传来源、目标岗位、渠道信息
- 上传后自动进入目标岗位待筛选
- 普通简历文件大小和文件头校验
- 简历解析失败可保留并重试
- 候选人详情支持手动编辑资料和技能
- 编辑后刷新相关岗位匹配
- 简历解析结构扩展到项目、证书、语言、意向城市等字段

核心文件：

- `backend/app/api/resume.py`
- `backend/app/api/candidates.py`
- `backend/app/services/resume_service.py`
- `base_agent/resume_parser.py`
- `frontend/src/features/candidates/pages/CandidateProfilePage.tsx`
- `frontend/src/features/candidates/pages/CandidatesPage.tsx`
- `frontend/src/pages/UploadPage.tsx`
- `frontend/tests/candidate_profile_*.test.mjs`
- `backend/tests/test_resume_parse_retry.py`
- `backend/tests/test_workflow_enhancements.py`

### 包 C：需求管理与业务反馈阶段

目的：让招聘需求、业务反馈、岗位进展形成管理闭环。

包含：

- 新增 `RecruitmentDemand` 模型
- 新增 `/api/demands` 后端接口

- 新增 `business_review` 阶段
- 需求风险标记：逾期、业务反馈待处理、推荐多但无面试、HR 侧无推荐
- 招聘专员只能看和操作自己负责范围内的需求

核心文件：

## 4. 打版前验证门槛

自动化验证必须全部通过：

```bash
./.venv312/bin/python -m pytest backend/tests base_agent/tests -q
cd frontend && npm run lint
cd frontend && npm run typecheck
cd frontend && npm run build
for f in frontend/tests/*.test.mjs; do node "$f" || exit $?; done
python3 -m pip_audit -r backend/requirements.txt
cd frontend && npm audit --audit-level=moderate
```

允许存在但要记录的 warning：

- 后端 SQLAlchemy `Query.get` legacy warning
- `datetime.utcnow` deprecation warning
- 测试环境 JWT secret 较短 warning
- 前端已有 React hook dependency warning
- Vite chunk 大小提示

## 5. 手工冒烟路径

打 `v0.1-internal` 前建议按真实用户点一遍：

1. 管理员登录，确认用户管理可用。
2. 招聘专员上传简历，带目标岗位和来源信息。
3. 招聘专员查看候选人库，筛选来源、城市、解析状态。
4. 招聘专员编辑候选人详情并刷新匹配。
5. 招聘专员创建岗位，运行匹配，批量加入流程。
6. Pipeline 推进到业务待反馈、一面、Offer。
7. 面试官查看分配并提交反馈。
8. 管理者查看首页提醒和 BI 看板。
9. AI 助手查询候选人、岗位、流程，并确认写操作不会越权。
10. 尝试越权路径：其他招聘专员看不到不属于自己的候选人/需求。

## 6. 暂缓到下一版

这些不建议塞进 `v0.1-internal`：

- 多租户隔离
- 对象存储替代本地 uploads
- 完整审计日志与导出
- 异步队列和大批量解析
- SQLAlchemy 2.x 写法整体迁移
- 全面清理前端 hook warning
- Vite 分包优化
- 生产级密钥和 HTTPS 部署强化

## 7. 建议提交顺序

建议按以下顺序分提交或分 PR：

1. `security: tighten auth and access control`
2. `resume: harden upload and candidate profile editing`
3. `demands: add recruitment demand workflow`
4. `bi: add manager alerts and funnel metric fixes`
5. `demo: add demo readiness cleanup`
6. `frontend: polish recruiting workflow screens`
7. `docs: add internal release notes`

如果某些文件同时包含多个包的改动，优先按“能通过测试、能讲清业务价值”的包拆，不为了完美拆分而冒险手工切碎。
