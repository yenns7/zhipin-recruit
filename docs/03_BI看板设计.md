# 03 BI 监控看板设计 — HireInsight（难点模块）

> 管理者视角监控每位 HR 专员的招聘工作内容与效能
> v1.0 ｜ 2026-06-13 ｜ 关联 [01 PRD](./01_PRD.md)、[02 改造实施计划](./02_改造实施计划.md)

---

## 1. 为什么是难点

原仓库（FindJobs-Agent）是单人工具，**零埋点、零多角色、零汇总**。要做团队效能看板，三样东西都得从头建：

| 难点 | 具体表现 |
|---|---|
| 数据源要自产 | 没有任何操作日志，每个 API 动作都得补埋点 |
| 口径要统一 | "有效简历""到场""通过"定义不一致，指标就失真 |
| 实时与历史并存 | 既要看当天进度，又要看历史趋势，需分层计算 |
| 考核敏感 | 用于评 HR 专员，指标设计不当会引发管理争议 |
| 下钻交互 | 全局 → 专员 → 岗位 → 候选人，逐层穿透 |

---

## 2. 谁看、想回答什么

| 使用者 | 核心问题 |
|---|---|
| HRD/总监 | 团队整体进度健康吗？瓶颈在哪个环节？哪个岗位卡住？ |
| 招聘主管 | 我的专员谁忙谁闲？谁转化率高/低？怎么分配岗位？ |
| HR 专员（自看） | 我手上岗位进度如何？离目标差多少？ |

---

## 3. 指标体系

### 3.1 招聘漏斗（流程效能）
分层：`JD发布 → 简历导入 → 有效简历 → AI预筛通过 → 部门面试 → Offer → 入职`

| 指标 | 公式 |
|---|---|
| 简历有效率 | 有效简历 / 导入总数 |
| AI 预筛通过率 | 预筛建议通过 / 预筛完成 |
| 面试到场率 | 实际到场 / 邀约 |
| 面试通过率 | 面试通过 / 面试数 |
| Offer 接受率 | 接受 / 发出 |
| 入职率 | 入职 / Offer |
| 整体转化率 | 入职 / 简历总数 |

### 3.2 周期（速度）
- 平均招聘周期（JD 发布→入职天数）
- 各 stage 平均停留时长（定位瓶颈）
- 简历响应时长（导入→首次处理）

### 3.3 HR 专员效能（本期重点）
| 指标 | 公式 | 维度 |
|---|---|---|
| 人均处理简历数 | 该专员处理简历总量 | 工作量 |
| 人均推进次数 | 状态流转次数 | 活跃度 |
| 人均发起预筛数 | 触发的 AI 面试数 | 工作量 |
| 专员转化率 | 其负责候选人的入职/简历比 | 质量 |
| 人均在招岗位数 | 当前活跃 JD 数 | 负荷 |
| 平均响应时长 | 处理简历及时性 | 效率 |

> ⚠️ 效能指标定位为"协助分配与改进"而非简单排名；口径需阶段 2 与业务方共定（假设 A4）。

## 4. 埋点设计（数据源）

BI 全部数据来自 `events` 表（PRD 模块 F2）。每个 API 写操作落一条事件：

| action | 触发 API | 字段 |
|---|---|---|
| `resume.uploaded` | `/api/resume/upload` | actor_id, count |
| `resume.parsed` | 解析 worker 完成 | candidate_id, valid |
| `interview.started` | `/api/interview/start` | candidate_id, job_id, actor_id |
| `interview.scored` | AI 出报告 | candidate_id, pass, score |
| `pipeline.moved` | Kanban 流转 | candidate_id, from, to, actor_id |
| `offer.sent`/`offer.accepted` | Offer 节点 | candidate_id, actor_id |
| `candidate.onboarded` | 入职 | candidate_id, actor_id |

**口径铁律**：每个漏斗指标必须能由 `events` 的某两个 action 计数相除推导，定义写入"指标字典"。

## 5. 看板布局（三层下钻）

### 第 1 层 团队总览（HRD 默认页）
```
┌────────────────────────────────────────────────────┐
│ KPI 卡：在招岗位 | 本月入职 | 整体转化率 | 平均周期    │
├─────────────────────────┬──────────────────────────┤
│ 招聘大漏斗（全团队）      │ 各 HR 专员效能横向条形对比  │
│ 发布→简历→预筛→面试→入职 │ (处理量 / 转化率 双轴)     │
├─────────────────────────┼──────────────────────────┤
│ 招聘周期趋势折线（按周）  │ 瓶颈环节热力（停留 TOP）   │
└─────────────────────────┴──────────────────────────┘
      点任一专员 → 下钻第 2 层
```

### 第 2 层 单专员详情
- 负责岗位列表 + 每岗进度
- 个人漏斗 vs 团队均值（高于/低于标色）
- 工作量明细：处理简历数、发起预筛数、流转次数、平均响应时长
- 近 N 周活跃度曲线
- 点任一岗位 → 第 3 层

### 第 3 层 单岗位/候选人明细
- 候选人在各 stage 分布（Kanban 镜像）
- 停留超阈值的候选人红色高亮
- 跳转候选人画像/预筛报告

## 6. 技术方案

### 6.1 数据分层
| 层 | 内容 | 技术 |
|---|---|---|
| 采集 | 埋点中间件写 `events` | Flask middleware |
| 明细 | `events` + 业务表 | PostgreSQL |
| 聚合 | 按天/专员/岗位预算 → `metrics_daily` | Celery beat 定时 ETL |
| 服务 | 读汇总表 + 当天明细补算 | Flask BI service |
| 展现 | 图表 + 下钻 | React + ECharts |

> 取舍：历史走预聚合（快稳），当天实时补算（准），避免每次全表扫描。

### 6.2 BI 接口
| 接口 | 返回 |
|---|---|
| `GET /api/bi/overview?range=` | 团队 KPI+漏斗+趋势 |
| `GET /api/bi/staff?range=` | 各专员效能列表（可排序） |
| `GET /api/bi/staff/{id}?range=` | 单专员详情+均值对比 |
| `GET /api/bi/job/{id}` | 单岗位候选人分布 |

权限：BI 接口仅主管/管理员可访问全局；专员仅能访问 `staff/{自己id}`（RBAC 校验）。

### 6.3 指标字典示例
```json
{
  "ai_pass_rate": {
    "name": "AI预筛通过率",
    "formula": "count(interview.scored where pass=true) / count(interview.scored)",
    "dimensions": ["hr_id", "job_id", "date"]
  }
}
```

## 7. 实施要点（对应 [02](./02_改造实施计划.md) 阶段 3）
1. 先定埋点 + 指标字典，再做看板（口径先行防返工）。
2. 先打通第 1 层端到端，再补第 2/3 层下钻。
3. 聚合表先 T+1，再视需要做近实时。
4. 效能指标上线前与业务方对齐"考核 vs 改进"定位（假设 A4）。

## 8. 风险提示
- 数据量小时趋势失真 → 设最小样本提示。
- 专员强制排名易引发内耗 → 默认"对比均值"而非"排名"。
- 埋点遗漏使指标偏低 → 关键路径埋点需单测覆盖。

> 指标公式与布局为设计建议，口径需阶段 2/3 与业务方共同定稿。
