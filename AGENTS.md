# Project Execution Rules

本文件是智聘项目的 AI/Codex 项目级执行规范。除非用户在当前会话中明确要求跳过某条规则，否则所有开发、修复、重构、配置调整和文档维护任务都默认遵循本文件。

`AGENTS.md` 是项目级执行规则，不是每次业务改动都要更新的业务文档。普通功能、页面、接口、BI 口径或运行配置变化时，应优先更新对应的产品文档、技术文档、运行文档或部署文档；只有当项目执行规范本身发生变化，或用户明确要求修改项目级规则时，才修改本文件。

## 文档同步规则

任何产品流程、接口字段、端口、BI 口径、AI 助手行为发生变化时，必须同步更新相关文档；如无需更新，最终回复必须说明原因。

相关文档包括但不限于：

- `README.md`
- `RUNNING.md`
- `DEPLOYMENT.md`
- `docs/` 下的 PRD、BI 设计、系统设计、试点上线检查清单等文档

除非变更对象是项目执行规范本身，否则不要把普通业务变更写进 `AGENTS.md`。

## 执行要求

- 改动前先判断本次变更是否影响产品流程、接口字段、端口、BI 口径或 AI 助手行为。
- 如果影响，必须同步修改对应文档。
- 如果不影响，也必须在最终回复里说明“本次无需同步文档”的原因。
- 最终回复必须说明改了什么、验证了什么、文档是否同步。
- 如果用户当前明确指令与本文件冲突，优先遵循用户当前明确指令，并在回复中说明冲突点。

## 试点 / 上线判断规则

当用户询问“是否可以试点上线”“是否已经相对完善”“是否可以发给真实用户测试”时，不能只按主流程能跑通、自动化测试通过或核心权限正常来判断。必须先区分：

- **技术可运行**：服务能启动、接口能通、自动化测试通过。
- **产品可试用**：不同角色能完成主要工作，遇到缺数据、误操作、空列表时也知道下一步怎么办。
- **生产可上线**：具备真实数据安全、备份、合规、审计、监控和运维保障。

在给出“可试点”“可上线”“相对完善”等结论前，必须完成并说明以下审查：

- **角色视角审查**：分别从招聘专员、招聘经理/负责人、面试官、管理员视角走关键路径。
- **缺口入口审查**：检查岗位、候选人、面试官、账号、权限、BI 口径、AI 助手写操作等入口在“没有可选项、数据为空、数据被关闭/停用、用户无权限”时是否有清楚提示和下一步入口。
- **反向操作审查**：检查关闭后能否恢复、推进后能否撤回或修正、负责人能否调整、错误数据能否补救；暂不支持的必须作为风险说明。
- **数据归属审查**：检查候选人负责人、流程推进人、面试反馈人、岗位归属和 BI 统计口径是否一致，避免“看得到但算不清”。
- **文档和限制审查**：检查运行方式、账号密码、试点说明、已知限制和下一步优化是否写清楚。

如果以上审查没有完成，最终回复不得笼统承诺“可以上线”或“已经很完善”，只能说“技术主流程可运行”或“仍需补产品试点检查”。

## 用户视角验收规则

涉及页面、流程、权限、数据看板或试点体验的改动，除自动化测试外，还必须用普通用户语言检查：

- 用户第一眼是否知道该做什么。
- 用户没有数据时是否知道去哪里创建或导入。
- 用户选不到岗位、候选人、面试官、账号时是否有明确原因和下一步。
- 用户误关闭、误推进、填错信息时是否有补救路径；如果没有，必须在风险里说明。
- 管理者看到的数据是否能解释“谁做了什么、推进了多少、卡在哪里、责任在谁”。
