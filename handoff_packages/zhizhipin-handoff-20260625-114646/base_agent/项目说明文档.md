## 第一章 问题：求职者的噩梦

### 1.1 光是字节一家，就够你看一天

2026 年春招季，打开字节跳动校招官网——

![字节跳动 2026 校招首页：开启新的工作（7635）](report/docs/bytedance_all_jobs.png)

**7,635 个在招岗位。** 好，搜索「后端开发」缩小范围？

![搜索「后端开发」之后：开启新的工作（1095）](report/docs/bytedance_backend_search.png)

**还有 1,095 个。** 而这 1,095 个岗位虽然都叫「后端开发」，技术栈却可能是 Java、Go、Python、C++ 中的任何一种，方向可能是广告系统、推荐引擎、基础架构、AI Agent……每个岗位要求千差万别。

**但字节只是开始。** 春招季求职者通常要同时关注 5-10 家公司：

| 公司 | 校招在招岗位数（2026 春招） |
|:-----|:---------------------------|
| 字节跳动 | 7,635 |
| 腾讯 | 4,000+ |
| 阿里巴巴 | 3,500+ |
| 百度 | 2,000+ |
| 美团 | 2,500+ |
| **合计** | **约 20,000+** |

一个想找后端开发工作的应届生，面对的是 **数万个岗位**、**数千个「后端开发」搜索结果**——而没有任何工具帮他回答最关键的问题：**「哪个岗位最适合我？」**

### 1.2 三大核心痛点

**痛点一：信息过载，无法精准筛选。** 搜索「后端开发」返回上千结果。招聘平台只能按地区、学历粗略筛选，不能按技术栈（只看 Go 相关）或匹配度排序。求职者只能花几个小时逐个点开岗位描述，肉眼判断是否合适。

**痛点二：不知道自己什么水平。** 简历写「熟悉 Python」，但到底是 2 分还是 5 分？投了不匹配的岗位 → 简历石沉大海；低估了自己 → 错过了本来够得着的好机会。没有客观、量化的技能评估，求职就是在盲投。

**痛点三：面试准备没有方向。** 拿到字节某个岗位的面试邀请，不知道这个岗位会考什么。网上搜「后端面试题」，出来的是通用题库，和具体岗位的技能要求完全脱节。准备方向错误 → 面试表现不佳 → 错失 offer。

### 1.3 我们的答案

**FindBestCareers** 用 AI 一次性解决这三个问题：

| 痛点 | 我们的解法 |
|:-----|:---------|
| 1,095 个岗位看不过来 | AI 评估你的技能 → 按匹配度排序 → **最适合的岗位排最前面** |
| 不知道自己什么水平 | 上传简历 → AI 对 80+ 技能打分（1-5 星）→ **精确量化你的能力画像** |
| 不知道面试考什么 | 选择目标岗位 → 点击「模拟面试」→ **AI 面试官针对该岗位出题** |

**从「花一整天逐个浏览」到「一分钟找到最匹配的岗位，三十分钟完成针对性面试训练」。**

## 第四章 AI 创新性

本系统的每一个环节都有深度的提示工程设计、结构化输出解析和质量保证机制，而非简单的 API 封装。

### 4.1 创新一：V4 技能评分框架（源自脉脉生产系统）

不是数关键词出现次数，而是一套有**强制规则**的评分体系：

- **基准分 3 分**（代表「熟练」），每项技能从这个基准出发调整
- **强制降分规则**（不可跳过）：仅辅助参与 → 2 分；仅提及无实证 → 1 分
- **严格升分规则**（门槛极高）：高级领导力 + 可量化影响 → 4 分；行业专家级 → 5 分
- **背景因素分析**：综合学历层次（博士 > 硕士 > 本科）、院校排名（全球 Top50/C9 > 985/211）、公司声望（FAANG/BAT > 知名公司）、工作年限

### 4.2 创新二：岗位强度信号检测

系统自动识别岗位的技术强度并动态调整评分基准：

- **5 种信号**：博士学位要求（权重 4）、旗舰人才计划（权重 3）、顶级会议 ACL/NeurIPS/CVPR（权重 3）、国际实验室（权重 2）、前沿大模型研究（权重 2）
- **4 个等级**：旗舰研究（基准 5）、研究（基准 4）、高级（基准 3）、标准（基准 3）
- **外部知识增强**：通过搜索引擎 API 检索实验室和项目背景信息，丰富 LLM 判断上下文

### 4.3 创新三：岗位针对性自适应面试（核心创新）

- **岗位绑定**：必须先选择具体岗位，所有问题围绕该岗位技能要求生成
- **三阶段状态机**：自我介绍 → 岗位技术问答（5 题）→ 综合反馈报告
- **自适应难度**：基于滚动平均分动态调节，高分自动加难、低分自动降难
- **五维度实时评估**：理解力、逻辑结构、方案可行性、细节深度、沟通表达
- **智能追问**：检测到回答薄弱环节时，自动生成针对性追问

### 4.4 创新四：多智能体协作架构

4 个专业化 Agent 通过统一 LLM 客户端协同工作：

| Agent | 职责 | 关键技术 |
|:------|:-----|:---------|
| 简历解析 Agent | PDF 提取 + 结构化信息识别 + V4 技能评分 | 多编码支持、LLM JSON 解析 |
| 岗位匹配 Agent | 语义匹配 + 模糊匹配 + 复合排序 | 覆盖率 + 水平加权公式 |
| 面试 Agent | 三阶段状态机 + 自适应问答 + 五维度评估 | 岗位绑定提示工程 |
| 统一 LLM 客户端 | 多供应商 + Key 轮换 + 重试恢复 | OpenAI/DeepSeek 双供应商 |

## 第五章 技术实现

### 5.1 技术栈

| 层级 | 技术 |
|:-----|:-----|
| 前端 | React 18 + TypeScript + Tailwind CSS + Vite |
| 后端 | Python + Flask（RESTful API） |
| AI 引擎 | GPT-4o（通过 OpenRouter，支持多供应商切换） |
| 数据 | 2,400+ 真实岗位、600+ 技能分类、100+ 职位类别 |

### 5.2 关键指标

| 指标 | 数值 |
|:-----|:-----|
| Python 后端代码量 | 4,700+ 行，跨 11 个模块 |
| React 前端代码量 | 1,000+ 行，5 个核心组件 |
| 技能分类体系规模 | 600+ 技能标签，覆盖 100+ 职位类别 |
| 岗位数据集规模 | 2,400+ 真实岗位（字节跳动 2026 校招） |
| 面试评估维度 | 5 维度 × 0-5 分制 |
| 并发处理能力 | 10 线程并行，API Key 自动轮换 |

### 5.3 系统架构

前端（React）通过 RESTful API 与 Flask 后端通信。后端将任务分发给 4 个专业化 Agent。所有 Agent 通过统一 LLM 客户端调用大语言模型，内置 API Key 轮换、指数退避重试、响应格式校验等生产级特性。

部署采用 Docker 容器化，Flask 同时服务 API 和前端静态文件，在 7860 端口对外提供服务。

## 第七章 商业模式与未来规划

### 7.1 商业模式

FindBestCareers 瞄准的是一个巨大的市场——中国每年有 **1,000 万+** 应届毕业生需要求职，加上社招人群，潜在用户规模达 **数千万级**。

**第一阶段：贴片广告变现**

产品上线初期，通过以下方式实现收入：

- 简历分析结果页 —— 底部贴片广告（如求职培训、简历模板）
- 岗位匹配结果页 —— 信息流中穿插广告
- 面试反馈报告页 —— 底部推荐广告（如面试辅导课程）

用户使用产品的三个核心页面，每一个都天然适合展示与求职相关的广告，**转化率远高于通用广告**。

**第二阶段：推荐位广告**

当平台流量达到一定规模后（日活 10 万+），开放付费推荐位：

- 企业可付费将自己的岗位提升到匹配结果的前列（类似百度推广的竞价排名）
- 培训机构可付费在面试反馈报告中推荐自己的课程
- 猎头公司可付费获取人才画像数据（脱敏后）

**这是一个自然的从 C 端流量到 B 端变现的商业闭环。**

### 7.2 未来规划

| 阶段 | 规划 | 预期效果 |
|:-----|:-----|:---------|
| 近期 | 基于真实面试数据微调面试模型 | 问题质量和评分准确性提升 30%+ |
| 近期 | 支持中英文双语面试 | 覆盖留学生和外企求职场景 |
| 中期 | 对接更多招聘平台（牛客、Boss 直聘等） | 岗位库从 2,400 扩展到 50,000+ |
| 中期 | 面试反馈报告 PDF 导出 | 用户可保存和分享面试记录 |
| 中期 | 上线贴片广告系统 | 实现首批商业收入 |
| 远期 | 用户系统 + 持久化数据库 | 历史记录、进度追踪、个性化推荐 |
