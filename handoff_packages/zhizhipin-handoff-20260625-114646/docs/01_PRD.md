## 1. 产品定位

智聘把 AI 简历解析、岗位画像、候选人匹配、AI 预筛面试和 BI 看板整合成招聘侧工作流。HR 批量导入简历，系统自动解析、打标、画像、为 JD 荐人、辅助面试评估；管理者用 BI 看板监控团队效能。

### 1.1 当前 MVP 实现边界

本文是产品范围真源；部分验收项是产品目标要求。当前实现真相以 [SDD](./SDD-智聘招聘系统-v1.0.md) 为准。

当前 MVP 已实现招聘闭环、核心 `org_id` 组织隔离、试点审计和轻量防重；简历上传、AI 解析、JD 结构化、AI 面试等仍以请求内同步处理为主。Redis/Celery 异步队列、大规模批量导入 worker、对象存储、前端组织管理后台、导出审批/水印、企业级不可变审计、OA 自动承接需求属于后续规划，不是当前一期必须补齐的前提。

## 2. 角色与权限

| 角色 | 代码角色 | 能做什么 | 数据可见范围 |
|---|---|---|---|
| 招聘专员 | `recruiter` | 传简历、看候选人画像、推进流水线、发起 AI 预筛 | 当前组织内，仅自己负责的候选人/岗位 |
| 招聘经理/负责人 | `manager` | 分配岗位、看 BI 看板、考核效能 | 当前组织内团队 |
| 面试官 | `interviewer` | 看推荐简历、填面试评价 | 被指派的候选人 |
| 管理员 | `admin` | 账号/权限、标签体系、模型配置 | 当前组织；一期不提供跨组织管理后台 |

> 旧称对应：HR 专员 → 招聘专员(recruiter)；招聘主管/HRD → 招聘经理/负责人(manager)；用人部门面试官 → 面试官(interviewer)；系统管理员 → 管理员(admin)。RBAC 边界以 `backend/app/middleware/auth.py` 和 `docs/SDD-智聘招聘系统-v1.0.md §5` 为准。

## 3. 功能模块与对应代码

### 模块 A：批量简历导入与处理 ★必备1

| 子功能 | 复用/新增 | 落到代码 |
|---|---|---|
| A1 批量上传（多文件/zip） | 扩展 | 当前 `POST /api/resume/upload` 收多文件/ZIP 并同步解析入库；大批量异步队列为后续规划 |
| A2 解析结构化 | 复用 | `resume_parser.py` 输出统一 JSON；批次级进度/失败重试 worker 为后续规划 |
| A3 打标评分 | 复用 | `tag_rate.py`，标签写 `candidate_tags`，兼容 `Python , 5 , AI` 格式 |
| A4 候选人画像 | 新增 | 聚合结构化字段+标签→候选人判断卡片；默认展示推荐判断、核心亮点、核心技能和待确认风险，雷达降级为辅助，全量技能标签折叠查看 |
| A5 来源渠道标准化 | 新增 | 上传批次记录候选人来源，支持 BOSS直聘、58同城、猎聘、鱼泡直聘等选项，供候选人库筛选和 BI 渠道质量统计；来源信息默认选填收起 |

**用户故事**：作为招聘专员，我拖拽上传一个含 80 份简历的 zip，系统先把解析成功的候选人沉淀到简历库。等我确认招聘需求后，再到简历库选择目标岗位 / 招聘需求查看适配候选人，并把合适的人加入该需求流程。

**验收**：上传页不要求选择岗位，上传成功后统一进入简历库；简历库顶部入口使用“岗位匹配”，筛选区使用“目标岗位 / 加入招聘需求”和“入需求流程状态”，避免和普通筛选重复；候选人来源、内推人/猎头联系人和批次备注为选填信息，默认收起；BI 中同一渠道不会因简称或别名被拆分。单批 ≥100 份、关键字段抽取准确率 ≥90%、失败项可单独重试、PII 字段脱敏存储属于产品验收目标，当前实现状态以 SDD 为准。

### 模块 B：岗位管理与人岗匹配 ★必备1

| 子功能 | 复用/新增 | 落到代码 |
|---|---|---|
| B1 JD 录入与结构化 | 复用 | `job_agent.py` 抽取学历/专业/技能要求 |
| B2 岗找人匹配 | 当前主线 | 输入 JD 画像，遍历当前账号权限范围内的简历池算匹配度，输出排序候选人+理由 |
| B3 智能荐人 | 新增 | 新 JD 发布自动从存量简历池召回（激活沉睡简历） |
| B4 岗位缺口引导与恢复 | 新增 | 关键流程无可选岗位时引导新建岗位；已关闭岗位可查看并恢复在招 |

**用户故事**：作为 HR，我录入一条"高级后端"JD，系统结构化出要求后，从我有权限查看的简历池返回按匹配度排序的候选人列表，每个标注"为什么匹配/缺什么"。

**验收**：招聘需求是主线，岗位/JD 是需求下的匹配画像；新建需求可直接填写岗位职责/任职要求并自动生成岗位画像，也可复用已有岗位画像。给定 JD 返回 Top-N 候选人 + 可解释匹配理由；工作台和需求卡片提供“匹配候选人”入口；岗位匹配页提供“AI 推荐 / 全部候选人”视角，HR 可在权限范围内按姓名、公司、岗位、学校、技能或邮箱搜索候选人，并按匹配度、入需求流程状态、匹配技能和缺失技能筛选；AI 没推荐到的人也能人工补找、查看当前岗位画像匹配预览并加入该需求流程；没有候选人时有上传简历入口，误关闭岗位画像后可恢复；下拉框有数据但找不到目标项时，也能看到新建需求、上传简历或联系管理员的入口。

### 模块 C：AI 预筛面试 ★必备1

基于 AI 面试能力，为招聘侧生成预筛题，并对提交的作答输出结构化评分和建议。

| 接口 | 当前定位 | 说明 |
|---|---|---|
| `/api/interview/start` | 发起 AI 预筛 | HR 对某候选人发起预筛，按 JD 出题 |
| `/api/interview/submit` | 提交作答并评分 | 当前由招聘侧录入/提交作答并触发 AI 评分；候选人自助异步作答入口为后续待确认能力 |

**输出**：预筛报告 = 分项评分 + 亮点 + 疑点（识别"包装简历"）+ 是否建议进真人面试。

**验收**：当前 MVP 中，招聘侧提交作答后 HR 收到结构化报告；若后续启用候选人自助异步作答，必须补充候选人入口、通知、授权告知、权限和合规验收。

### 模块 D：招聘流水线 ★新增

Kanban：`待初筛 → AI预筛 → 部门面试 → Offer → 入职 / 淘汰`。每次流转写 `events` 表（供 BI）。

**用户故事**：作为 HR，我把通过 AI 预筛的候选人卡片拖到"部门面试"，系统记录流转时间与操作人。

**误操作补救**：候选人推进错阶段或误淘汰时，HR/经理/管理员在管道里使用“修正阶段”，必须填写修正原因。系统追加一条带“阶段修正”标记的流水，当前阶段和 BI 当前存量按最新阶段计算，历史记录不删除。

**面试反馈归口**：面试阶段的人工反馈统一在“面试任务”页填写，但该页面不作为招聘专员、经理或管理员的左侧一级入口。候选人流程和工作台待办保留跳转入口并携带岗位/候选人参数，面试任务页自动定位到待反馈项；面试官在“我的面试”任务卡点击“填写反馈”后，页面自动切到待处理并滚动到对应反馈表。AI 预筛只作为辅助参考，不作为人工安排和反馈的主入口。

**流程直达**：招聘专员工作台“今日待办”可直接进入候选人流程的业务待反馈、面试中、Offer 等阶段；工作台 KPI 数字也可点击下钻到候选人库、招聘管理或对应阶段流程。候选人流程默认展示有候选人且更需要处理的阶段，空阶段必须提供去非空阶段、去匹配更多候选人或上传简历的下一步入口。

**需求补救**：用人需求误关闭、误暂停或业务重新确认继续招聘时，HR/经理/管理员可恢复需求，系统同步把岗位画像恢复为在招；需求优先级可在 A/B/C 之间重新调整，并记录调整原因，避免误降级后只能重新建需求。关闭、恢复和调整优先级必须通过站内确认弹窗填写业务原因，并说明这次操作会影响活跃需求、岗位状态或后续复盘。候选人更适合其他需求时，流程详情支持“转入其他招聘需求”，必须选择目标需求并填写原因；系统保留当前需求历史，把目标需求作为新的当前流程继续推进。

**关闭岗位冻结**：岗位关闭后只允许查看历史、受控导出和恢复在招；上传候选人到该岗位、推进流程、发 Offer、安排/提交面试、AI 预筛写回和 AI 助手改状态都必须由后端拒绝。

### 模块 E：BI 监控看板 ★必备2（难点，详见 [03](./03_BI看板设计.md)）

管理者看团队当前阶段分布、各专员效能对比、渠道质量、瓶颈环节和数据质量提醒，可三层下钻。周期指标按“本期入库简历 cohort”统计，当前阶段分布按全量最新阶段统计；当前流程人数不含已入职/已淘汰，入职占比用全流程去重人数做分母。

**验收**：主管/管理员可看当前组织内全局 BI；招聘专员只能看自己负责的岗位和候选人，不能通过协作候选人或修改 `job_id` 查看别人岗位漏斗；面试官不开放岗位级或专员级 BI。专员绩效统一按 `candidates.owner_hr_id` 归属，团队均值采用总入职/总简历的加权口径，来源渠道按标准渠道聚合，Offer 数高于面试进入数等异常要在总览和专员下钻显示数据质量提醒。

BI 页面必须用白话解释协同归属：HR 负责候选人推进，推进人只看操作留痕，面试官负责反馈闭环，用人部门负责岗位协同；异常数据旁给出下一步动作建议。

**AI 助手示例**：AI 助手首页示例问题按角色展示。招聘专员看到自己负责候选人的卡点和待反馈问题，经理/负责人看到团队漏斗和专员推进问题，管理员看到审计、权限和 AI 边界问题，避免用户被引导到无权限入口。

### 模块 F：系统支撑 ★新增

- F1 RBAC（角色权限隔离）
- F2 多组织隔离：核心招聘表、BI、通知、AI 工具和审计事件按 `org_id` 过滤；一期组织开通由部署初始化完成。这里的“多组织隔离”不等于 SaaS 式多租户运营后台；前端组织管理、自助开通和跨组织运营管理为后续规划。
- F3 操作埋点（写 `events`，BI 数据源）
- F4 PII 脱敏 + 访问审计 + 候选人授权告知；候选人详情查看、简历详情查看、候选人 CSV 导出、删除、AI 写操作和越权 403 进入试点审计日志；轻量试点不做导出审批/水印，但同一账号短时间高频导出会在管理员审计页标红；候选人删除走软删除、匿名化并移除原简历文件
- F5 轻量试点防重：重复上传同一批简历、重复推进同一阶段、重复安排同一面试、重复提交同一轮反馈时，后端复用已有结果；普通 JSON/表单写接口支持 `Idempotency-Key`，避免连点、网络超时重试或 webhook 重放制造重复业务数据
- F6 上传与误导入恢复：上传支持 PDF、DOCX、ZIP，旧版 `.doc` 因宏风险跳过；误导入可按上传批次撤回，撤回会匿名化候选人、删除原文件并写审计
- F7 候选人负责人转派：经理/管理员通过招聘专员下拉框选择新负责人，填写转派原因，不让业务用户输入用户 ID。

## 5. 端到端验收场景

1. HR 上传一批简历 → 当前 MVP 在请求内完成解析+打标+画像；100 份级大批量异步处理作为后续性能增强验收。
2. 录入 JD → 返回排序候选人列表 + 匹配理由。
3. 候选人完成 AI 预筛 → HR 收到评估报告。
4. 候选人在 Kanban 推进 → 全程留痕。
5. 管理者在 BI 看板看各专员漏斗与效能并下钻。
6. 专员只能看自己候选人；PII 脱敏、审计日志可查。
