# 03 BI 监控看板设计 — 智聘招聘系统（BI 口径真源）

> 管理者视角监控每位 HR 专员的招聘工作内容与效能
> v1.0 ｜ 2026-06-13 ｜ 关联 [01 PRD](./01_PRD.md)

---

## 0. 当前使用说明

本文是 BI 指标口径和责任归属真源。涉及“怎么算、算给谁、按什么时间范围算”，优先以本文为准。

角色名按 `docs/README.md` 的“角色名口径”理解：本文里的 HRD、招聘主管、用人部门负责人在 BI 查看和协同语境下均属于 `manager` 业务视角；HR 专员等同招聘专员（`recruiter`）；系统管理员等同管理员（`admin`）。这些是业务叫法，不代表新增技术角色。

本文中出现的 `metrics_daily`、Celery beat、T+1 ETL、ECharts 等，只能理解为后续扩展方向，不代表当前 MVP 已实现。当前实现以 [SDD](./SDD-智聘招聘系统-v1.0.md) 为准：BI 主要从 `pipeline_stages`、`candidates.owner_hr_id`、`interview_assignments`、`interview_feedback`、`jobs.department` 和 `events` 等业务表实时或请求内计算，前端沿用当前 React 图表栈。

## 1. 为什么是难点

BI 难点不在画图，而在招聘过程要能留下可解释的数据。要做团队效能看板，三样东西必须明确：

| 难点 | 具体表现 |
|---|---|
| 数据源要可靠 | 关键 API 动作要有业务流水或审计事件，不能靠前端临时拼数字 |
| 口径要统一 | "有效简历""到场""通过"定义不一致，指标就失真 |
| 实时与历史并存 | 既要看当天进度，又要看历史趋势，需分层计算 |
| 考核敏感 | 用于评 HR 专员，指标设计不当会引发管理争议 |
| 下钻交互 | 全局 → 专员 → 岗位 → 候选人，逐层穿透 |

## 2. 谁看、想回答什么

| 使用者 | 核心问题 |
|---|---|
| HRD/总监 | 团队整体进度健康吗？瓶颈在哪个环节？哪个岗位卡住？ |
| 招聘主管 | 我的专员谁忙谁闲？谁转化率高/低？怎么分配岗位？ |
| HR 专员（自看） | 我手上岗位进度如何？离目标差多少？ |
| 用人部门负责人 | 本部门面试反馈是否及时？哪些面试官通过/拒绝口径波动大？ |

## 3. 指标体系

### 3.1 招聘漏斗（流程效能）
分层：`JD发布 → 简历导入 → 有效简历 → AI预筛通过 → 部门面试 → Offer → 入职`

| 指标 | 公式 |
|---|---|
| 简历有效率 | 有效简历 / 导入总数 |
| AI 预筛通过率 | 预筛建议通过 / 预筛完成 |
| 面试到场率 | 实际到场 / 邀约 |
| 面试通过率 | 面试通过 / 面试数 |
| Offer 接受率 | 接受 / 发出 |
| 入职率 | 入职 / Offer |
| 整体转化率 | 入职 / 简历总数 |

### 3.2 周期（速度）
- 平均招聘周期（JD 发布→入职天数）
- 各 stage 平均停留时长（定位瓶颈）
- 简历响应时长（导入→首次处理）

### 3.3 HR 专员效能（本期重点）
| 指标 | 公式 | 维度 |
|---|---|---|
| 人均处理简历数 | 该专员负责且本期入库的简历总量 | 工作量 |
| 人均推进次数 | 状态流转次数，按 `pipeline_stages.updated_by` 统计 | 活跃度动作指标 |
| 人均发起预筛数 | 触发的 AI 面试数 | 工作量 |
| 专员转化率 | 本期入库且归属该专员的候选人入职/简历比 | 质量 |
| 人均在招岗位数 | 当前活跃 JD 数 | 负荷 |
| 平均响应时长 | 处理简历及时性 | 效率 |

> ⚠️ 效能指标定位为"协助分配与改进"而非简单排名；如要把指标用于正式考核，需要先和业务负责人确认口径。

### 3.4 面试责任归因（保护 HR 绩效）

主流程仍保持 `待筛选 → AI 初筛 → 业务待反馈 → 面试中 → Offer → 已入职 / 淘汰`，不把一面、二面、三面重新做成主流程阶段。

但 BI 需要在"面试中"下面记录更细的面试事实，用来区分：

- HR 是否已经推荐并推进了候选人；
- 用人部门是否及时反馈；
- 面试官是否通过/拒绝候选人；
- 不同轮次是否出现前后标准不一致。

| 维度 | 指标 | 口径 |
|---|---|---|
| 面试反馈跟进 | 面试安排数 | `InterviewAssignment` 中该面试官的安排数 |
| 面试反馈跟进 | 已反馈数 | `InterviewFeedback` 中该面试官提交的反馈数 |
| 面试反馈跟进 | 通过数 / 拒绝数 | `InterviewFeedback.passed = true / false` |
| 面试反馈跟进 | 待补反馈 / 超时反馈 | 已安排且面试时间已过，但无对应反馈 |
| 部门协同情况 | 部门面试安排数 | 按岗位 `department` 聚合面试安排 |
| 部门协同情况 | 部门通过数 / 拒绝数 | 按岗位部门聚合面试反馈 |
| 部门协同情况 | 参与面试官数 | 该部门相关面试记录中去重面试官数 |
| 轮次明细 | 第 1 轮 / 第 2 轮 / 第 3 轮 / 加面 / HR 面 / 业务面 / 技术面 | 作为面试记录维度，不作为管道主流程 |

产品解释：

> 如果 HR 推荐了候选人并推进到面试，但用人部门反馈慢、面试官之间标准不一致，BI 应能把责任链拆出来，而不是只用最终 Offer/入职结果评价 HR。

### 3.5 当前 MVP 对外展示口径

当前 MVP 不再把一面、二面、终面作为候选人流程主阶段对外考核，而是用更稳定的通用口径展示：

| 指标 | 当前口径 | 主要数据源 |
|---|---|---|
| 当前阶段分布 | 每个 `候选人 + 岗位` 只取全量最新阶段，不受 `days` 周期筛选影响 | `pipeline_stages` 最新行 |
| 当前流程人数 | 当前阶段为非终态的人数，只包含待筛选、AI 初筛、业务待反馈、面试中、Offer | `pipeline_total` |
| 归档人数 | 当前阶段为已入职或已淘汰的人数 | `archived_total = onboarded + rejected` |
| 全流程入职占比 | 已入职 /（当前流程人数 + 归档人数） | `conversion_rate = onboarded / funnel_total` |
| 有效推荐 | 候选人已进入招聘需求流程 | `pipeline_stages` 按候选人+岗位画像去重 |
| 推荐成功面试 | 候选人进入 `面试中` 阶段 | `pipeline_stages.stage in interview*` 统一归为 `interview` |
| 面试通过 | 面试反馈里标记通过的人 | `interview_feedback.passed = true` |
| Offer | 候选人进入 Offer 阶段 | `pipeline_stages.stage = offer` |
| 已入职 | 候选人进入已入职阶段 | `pipeline_stages.stage = onboarded` |
| 待补反馈 | 已安排且面试时间已过，同一候选人/岗位/轮次没有任何反馈；若替补面试官已提交同轮反馈，则原面试官不再算待补 | `interview_assignments` 对比 `interview_feedback` |

前端对外展示时用更白话的说明：有效推荐 = 已进入招聘需求流程；推荐成功面试 = 进入“面试中”阶段，不固定看一面/二面/三面；面试通过 = 面试反馈里标记通过的人；Offer = 已推进到 Offer 阶段；已入职 = 已推进到已入职阶段；主绩效按候选人负责人归属。

周期选择器只接受 `days=7/30/90`。非法值或空值按默认周期回退：团队总览和专员详情默认 30 天，单岗位 BI 默认 90 天。`days` 只影响“本期入库简历 cohort”类指标，例如 `staff`、`source_quality`、`resumes`、面试责任归因；不影响当前阶段分布、需求健康和管理提醒这类当前状态指标。

招聘专员绩效、专员详情 `performance`、专员详情漏斗统一按 `candidates.owner_hr_id` 归属；`pipeline_stages.updated_by` 只表示谁做了推进动作，适合后续做动作审计或推进动作榜，不作为主绩效归属。

团队均转化率采用加权口径：团队总入职数 / 团队总入库简历数，而不是把每个专员的个人转化率做简单平均，避免小样本专员把团队均值拉偏。

需求健康中的“HR 无推荐”只认可开放需求下仍处于非终态的当前推荐。当前技术实现仍通过 `recruitment_demands.job_id` 绑定岗位画像来计算需求流程；业务展示统一说“招聘需求流程”。如果某需求曾推荐过候选人但最新阶段已是已淘汰或已入职，则该需求不再被当作“仍有活跃推荐”。

候选人从一个招聘需求转入另一个招聘需求时，系统会在原需求追加一条 `rejected` 流水并写明“转入其他招聘需求”的原因，在目标需求追加一条 `pending` 流水。BI 当前存量按每个需求下候选人的最新阶段计算，所以原需求不再把该候选人计入活跃流程，目标需求会从待筛选开始承接后续推进。

前端 BI 看板首屏优先展示业务数字，标题区只保留“看团队招聘进度、卡点和协同跟进”短说明和周期筛选，不再放“怎么看数据”入口。详细指标口径和协同归属沉淀在本文档，页面内用数据质量提醒引导主管回业务页面核对。招聘专员工作台展示“我的本月业绩”和“今日待办”，首屏 KPI 支持点击下钻到候选人库、招聘管理或对应阶段流程。两处使用同一套 BI 绩效字段，避免专员自看和主管看板口径不一致。

### 3.6 来源渠道口径

简历上传页的 `来源渠道` 使用标准可选项，优先由 HR 点击选择，减少自由输入导致的 BI 拆分。

当前标准渠道：

`BOSS直聘`、`58同城`、`猎聘`、`鱼泡直聘`、`智联招聘`、`前程无忧`、`内推`、`官网`、`LinkedIn`、`其他`。

入库时会对常见别名做归一，例如：

- `BOSS`、`Boss直聘` → `BOSS直聘`
- `58`、`五八同城` → `58同城`
- `鱼泡`、`鱼泡招聘` → `鱼泡直聘`
- `领英`、`linkedin` → `LinkedIn`

BI 的 `source_quality` 使用归一后的 `UploadBatch.source_channel` 聚合，避免同一渠道因大小写、简称或中文别名被拆成多行。候选人库的来源筛选也复用同一标准渠道列表，并兼容历史别名数据。

渠道质量采用“简历入库 cohort”口径：先按 `Candidate.created_at >= now - days` 确定本期入库简历，再统计这些简历后续是否进入面试、Offer、入职。老简历在本期发生面试或 Offer，不会混入本期来源渠道分母，避免渠道转化率被动作周期冲高或冲低。

## 4. 埋点设计（数据源）

BI 数据来自 `events` 表和核心业务明细表。`events` 用于记录关键写操作；涉及当前阶段、面试轮次、面试官反馈、部门协同时，以业务表为准，避免仅靠事件表导致口径不完整。

每个 API 写操作落一条事件：

| action | 触发 API | 字段 |
|---|---|---|
| `resume.uploaded` | `/api/resume/upload` | actor_id, count |
| `resume.parsed` | 简历解析完成 | candidate_id, valid |
| `interview.started` | `/api/interview/start` | candidate_id, job_id, actor_id |
| `interview.scored` | AI 出报告 | candidate_id, pass, score |
| `pipeline.moved` | Kanban 流转 | candidate_id, from, to, actor_id |
| `offer.sent`/`offer.accepted` | Offer 节点 | candidate_id, actor_id |
| `candidate.onboarded` | 入职 | candidate_id, actor_id |

面试责任归因额外读取：

| 表 | 用途 |
|---|---|
| `interview_assignments` | 面试安排、面试官、轮次、计划时间、待补反馈 |
| `interview_feedback` | 面试反馈、通过/拒绝、评分、面试官、原因分类 |
| `jobs` | 岗位与用人部门归属 |

**口径铁律**：每个指标必须能追溯到明确的业务事实表或事件；涉及考核的指标必须在指标字典中写清楚分子、分母、归属人和时间范围。

### 4.1 账号归属口径

MVP 试用阶段建议招聘专员一人一个内部试用账号。账号名可以是中性的 `hr01@mvp.local`、`hr02@mvp.local`，但不要多人长期共用同一个账号。

BI 看板不是按“账号邮箱文字”统计，而是按系统里的用户 ID 统计。常见归属如下：

- 候选人负责人：看 `candidates.owner_hr_id`。
- 岗位负责人：看 `jobs.owner_hr_id`。
- 流程推进人：看 `pipeline_stages.updated_by`。
- 面试反馈人：看 `interview_feedback.interviewer_id`。
- 上传与系统事件：看事件或业务表里记录的操作者。

主 BI 绩效默认回答“这个候选人属于谁负责”，所以看 `candidates.owner_hr_id`；动作审计回答“这一步是谁点的”，才看 `pipeline_stages.updated_by`。两者不要混用，否则主管总览和专员详情会出现同一候选人归属不一致。

所以，一人一个账号的价值是：后续能看清每个招聘专员推进了多少候选人、推荐进入面试多少人、面试通过多少人、转 Offer 多少人。多人共用账号时，BI 只能统计这个共用账号的总动作，无法拆出个人贡献。

BI 协同归属说明沉淀在本文档，面向主管用白话说明：

- HR 负责候选人推进：看候选人负责人，适合评估推荐、推进和入职结果。
- 面试官负责反馈闭环：看谁被安排面试、谁提交反馈，待补反馈用于提醒跟进。
- 用人部门负责岗位协同：按岗位部门汇总安排、通过/拒绝和反馈滞留，帮助判断是否需要协调部门。
- 推进人只看操作留痕：最后推进人用于查谁点了阶段，不直接当 HR 绩效。
- 如果数字异常：先看数据质量提醒，再去候选人流程、面试任务页或跳阶记录里核对。

候选人负责人转派后，后续绩效归新负责人；历史推进人、面试反馈人和事件流水不重写。误推进后的“修正阶段”同样只追加新流水，当前阶段分布和当前存量按最新阶段计算，历史错误推进记录保留供审计和复盘。

## 5. 看板布局（三层下钻）

### 第 1 层 团队总览（HRD 默认页）
```
┌────────────────────────────────────────────────────┐
│ KPI 卡：在招专员 | 当前入职 | 全流程入职占比 | 当前流程人数 │
├─────────────────────────┬──────────────────────────┤
│ 招聘大漏斗（全团队）      │ 各 HR 专员效能横向条形对比  │
│ 发布→简历→预筛→面试→入职 │ (处理量 / 转化率 双轴)     │
├─────────────────────────┼──────────────────────────┤
│ 招聘周期趋势折线（按周）  │ 瓶颈环节热力（停留 TOP）   │
│ 面试反馈跟进              │ 部门协同情况               │
└─────────────────────────┴──────────────────────────┘
      点任一专员 → 下钻第 2 层
```

### 第 2 层 单专员详情
- 负责岗位列表 + 每岗进度
- 个人漏斗 vs 团队均值（高于/低于标色）
- 工作量明细：处理简历数、发起预筛数、流转次数、平均响应时长
- 近 N 周活跃度曲线
- 点任一岗位 → 第 3 层

### 第 3 层 单岗位/候选人明细
- 候选人在各 stage 分布（Kanban 镜像）
- 停留超阈值的候选人红色高亮
- 跳转候选人画像/预筛报告

## 6. 技术方案

### 6.1 数据分层
| 层 | 内容 | 技术 |
|---|---|---|
| 采集 | 埋点中间件写 `events` | Flask middleware |
| 明细 | `events` + `pipeline_stages` + 候选人/岗位/面试业务表 | SQLite（开发/演示）/ PostgreSQL（生产） |
| 当前服务 | 请求时按最新阶段、负责人和面试事实表计算 | Flask BI API |
| 当前展现 | 图表 + 下钻 + 数据质量提醒 | React + 当前前端图表栈 |
| 后续聚合 | 按天/专员/岗位预聚合到 `metrics_daily` | 后续可引入 Celery beat / 定时 ETL |

> 当前取舍：先保证口径清楚、权限正确和可反查；数据量变大后，再引入预聚合、定时 ETL 或搜索/分析索引。不要为了提前做目标架构，直接在当前一期引入 Redis/Celery 或第二套 BI 存储。

### 6.2 BI 接口
| 接口 | 返回 |
|---|---|
| `GET /api/bi/overview?days=` | 团队 KPI+漏斗+专员效能+责任归因 |
| `GET /api/bi/staff/{id}?days=` | 单专员漏斗 + `performance` 个人绩效行 |
| `GET /api/bi/job/{id}?days=` | 单岗位候选人分布；招聘专员仅岗位负责人可访问，不能查看别人岗位的漏斗 |

所有 BI 漏斗 `funnel` 统一包含：

| 字段 | 含义 |
|---|---|
| `pending` / `ai_screen` / `business_review` / `interview` / `offer` | 当前处于各非终态阶段的人数，历史一面/二面/终面统一归入 `interview` |
| `onboarded` / `rejected` | 当前已归档到入职或淘汰的人数 |
| `pipeline_total` | 当前活跃流程人数，不含 `onboarded` 和 `rejected` |
| `archived_total` | 已归档人数，即 `onboarded + rejected` |
| `funnel_total` | 全流程去重人数，即 `pipeline_total + archived_total` |
| `conversion_rate` | `onboarded / funnel_total` |

`GET /api/bi/overview` 当前还返回：

| 字段 | 含义 |
|---|---|
| `staff` | 各招聘专员公开绩效行：按本期入库且归属该专员的简历 cohort 统计有效推荐、推荐成功面试、面试通过、Offer、入职、待补反馈等 |
| `source_quality` | 按标准来源渠道和本期入库简历 cohort 聚合的简历量、推荐成功面试、面试通过、Offer、入职率等 |
| `interviewer_accountability` | 按面试官聚合的安排数、反馈数、通过数、拒绝数、待补反馈、通过率 |
| `department_accountability` | 按用人部门聚合的安排数、反馈数、通过数、拒绝数、待补反馈、参与面试官数、轮次明细 |
| `data_quality_warnings` | 分子大于分母等数据异常提醒，例如 Offer 数高于面试进入数 |

`GET /api/bi/staff/{id}` 当前返回：

| 字段 | 含义 |
|---|---|
| `hr_id` | 招聘专员用户 ID |
| `funnel` | 该专员负责候选人的当前阶段漏斗，按 `candidates.owner_hr_id` 归属 |
| `performance` | 与主管 BI `staff` 行一致的个人公开绩效字段，用于招聘专员工作台自看 |
| `data_quality_warnings` | 该专员绩效行的数据异常提醒 |

`GET /api/bi/job/{id}` 当前返回：

| 字段 | 含义 |
|---|---|
| `job_id` / `job_title` | 岗位 ID 与岗位名称 |
| `scope` | 当前仅返回 `all`；表示当前账号可看该岗位完整漏斗 |
| `funnel` | 单岗位当前阶段漏斗，字段同上 |

`InterviewFeedback.reason_tags` 已作为结构化原因分类进入数据流。MVP 当前先用于面试反馈留痕，常用标签覆盖候选人原因、能力原因、协同原因和用人部门原因，例如“候选人已接受其他机会”“面试时间无法协调”“岗位暂停招聘”“部门内部意见不一致”。后续可在 BI 中扩展为“拒绝原因分布”“岗位要求变化”“部门内部意见不一致”“HC 暂缓或冻结”等责任归因统计。

权限：所有 BI 查询都先按当前登录用户的 `org_id` 隔离。`overview` 仅主管/管理员可访问当前组织全局；`staff/{id}` 允许主管/管理员访问本组织任意专员，招聘专员仅能访问自己，面试官不可访问；`job/{id}` 允许主管/管理员访问本组织全部岗位，岗位负责人招聘专员可看该岗位完整漏斗，非岗位负责人招聘专员和面试官不可访问岗位级 BI。

## 7. 实施要点
1. 先定埋点 + 指标字典，再做看板（口径先行防返工）。
2. 先打通第 1 层端到端，再补第 2/3 层下钻。
3. 当前 MVP 优先使用业务事实表实时计算；聚合表、T+1 ETL 和近实时补算属于后续性能增强。
4. 效能指标上线前与业务方对齐"考核 vs 改进"定位。

## 8. 风险提示
- 数据量小时趋势失真 → 设最小样本提示。
- 专员强制排名易引发内耗 → 默认"对比均值"而非"排名"。
- 埋点遗漏使指标偏低 → 关键路径埋点需单测覆盖。

> 指标公式和归属口径是当前 BI 真源；页面布局和预聚合技术方案可随实现阶段迭代。
