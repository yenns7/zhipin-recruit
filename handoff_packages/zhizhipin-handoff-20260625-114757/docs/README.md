## 先说结论

智聘当前是一套面向 HR、招聘负责人、面试官和管理员的 AI 招聘管理系统。主线是在现有 `frontend/` + `backend/` + `base_agent/` 上做前后端分离开发、单体部署收口。

当前开发和审查时请先分清两类文档：

- **执行规则**：AI/Codex 必须怎么读文档、怎么改、怎么验收。
- **当前真源**：当前产品、实现、BI、运行和部署到底以什么为准。

给下一任接手者的简单判断：

- 想知道“项目是做什么的”，先看根目录 `README.md`。
- 想知道“现在代码实际做到哪一步、改哪里风险大”，看 `SDD-智聘招聘系统-v1.0.md`。
- 想知道“本地怎么跑、试用账号和演示数据怎么用”，看 `RUNNING.md`。
- 想知道“生产/试点怎么部署、上线前缺什么”，看 `DEPLOYMENT.md`、`06`、`07`；这些清单不是完成证明，必须现场复核。
- 不需要读历史方案或旧计划；当前开发只按下面的真源文档和真实代码判断。

## 当前关键口径

### 架构口径

- 当前主线：`frontend/` React + Vite + TypeScript，`backend/` Flask + SQLAlchemy + JWT/RBAC，`base_agent/` 复用 AI 能力。
- 部署口径：开发期前后端分离联调；生产/试点收口为 Flask 单端口托管 `frontend/dist` + `/api/*`。
- 不要把项目带向微服务、微前端、第二套前端框架、第二套权限体系或绕开现有 `/api` 边界的并行运行面。

### 当前实现 vs 后续规划

当前 MVP 已实现招聘闭环、核心权限、核心 `org_id` 组织隔离、试点审计、需求/岗位/候选人/流程/面试/BI/AI 助手等能力。

以下是后续规划或生产增强，不是当前必须补齐的开发前提：

- Redis/Celery 异步队列
- 大规模批量导入 worker
- 对象存储替代本地 uploads
- 前端组织管理后台 / SaaS 式多租户运营能力
- 企业级不可变审计、导出审批、水印、字段级权限
- OA 自动承接需求、外部日历、webhook、外部通知真实发送

### 多组织 / 多租户口径

当前一期已做核心 `org_id` 隔离：岗位、候选人、流程、面试、BI、AI 工具和审计日志按当前组织过滤。

这里说的“不做 SaaS 式多租户”是指：一期不做前端组织管理后台、租户自助开通、跨组织运营管理、计费/套餐等 SaaS 能力；不代表没有组织级数据隔离。

### 角色名口径

权限判断只认后端技术角色枚举，不按中文叫法临时新增角色。后续文档和页面文案优先使用“标准中文名”；旧称和业务别名只按同一角色理解。

| 技术角色 | 标准中文名 | 可接受旧称/业务别名 | 说明 |
|---|---|---|---|
| `admin` | 管理员 | 系统管理员 | 当前组织内账号、审计、AI 边界和管理能力 |
| `manager` | 招聘经理/负责人 | 招聘主管、HRD、主管、负责人、用人部门负责人（看 BI/协同时） | 当前组织内团队视角；不要拆成第二套 manager/lead 权限 |
| `recruiter` | 招聘专员 | HR 专员、HR | 负责自己的候选人、岗位、需求流程和个人数据 |
| `interviewer` | 面试官 | 用人部门面试官 | 只处理分配给自己的面试和反馈 |

试用账号里的 `manager01@mvp.local` 和 `lead01@mvp.local` 都按 `manager` 权限理解；“招聘负责人”只是业务展示名，不是新的技术角色。
