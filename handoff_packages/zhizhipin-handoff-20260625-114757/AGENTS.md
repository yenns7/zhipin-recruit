# Project Execution Rules

本文件是智聘项目的 AI/Codex 项目级执行规范。除非用户在当前会话中明确要求跳过某条规则，否则所有开发、修复、重构、配置调整和文档维护任务都默认遵循本文件。

`AGENTS.md` 是项目级执行规则，不是每次业务改动都要更新的业务文档。普通功能、页面、接口、BI 口径或运行配置变化时，应优先更新对应的产品文档、技术文档、运行文档或部署文档；只有当项目执行规范本身发生变化，或用户明确要求修改项目级规则时，才修改本文件。

## 文档同步规则

任何产品流程、接口字段、端口、BI 口径、AI 助手行为发生变化时，必须同步更新相关文档；如无需更新，最终回复必须说明原因。

相关文档包括但不限于：

- `README.md`
- `RUNNING.md`
- `DEPLOYMENT.md`
- `docs/` 下的 PRD、BI 设计、系统设计、试点上线检查清单等文档

除非变更对象是项目执行规范本身，否则不要把普通业务变更写进 `AGENTS.md`。

## 动手前文档入口

AI 开始开发、修复、重构、配置调整或文档维护前，应先读取 `AGENTS.md` 和 `docs/README.md`；其中 `docs/README.md` 作为文档地图，用于判断本轮需要继续读取的最小必要文档集。

后续只读取与本轮任务直接相关的产品、技术、BI、运行、部署或试点文档；历史计划、报告和归档材料仅在任务明确相关时读取。

## 执行要求

- 改动前先判断本次变更是否影响产品流程、接口字段、端口、BI 口径或 AI 助手行为。
- 如果影响，必须同步修改对应文档。
- 如果不影响，也必须在最终回复里说明“本次无需同步文档”的原因。
- 最终回复必须说明改了什么、验证了什么、文档是否同步。
- 如果用户当前明确指令与本文件冲突，优先遵循用户当前明确指令，并在回复中说明冲突点。

## 最终回复固定要求

无论本轮是代码改动、文档改动、只读审查、方案判断还是问答解释，最终回复都必须覆盖：

- **本轮处理**：说明实际改了什么；如果未改文件，明确说明只是审查或判断。
- **验证依据**：说明实际运行的命令、查看的文件、使用的测试或形成判断的证据；如果未运行测试，说明原因。
- **文档同步**：说明已同步哪些文档；如果无需同步，说明本轮没有改变产品流程、接口字段、端口、BI 口径或 AI 助手行为等原因。

## 架构边界与 Owner 规则

当前项目主线是前后端分离开发、单体部署收口的招聘管理系统：

- 前端主线：`frontend/`，React + Vite + TypeScript。
- 后端主线：`backend/`，Flask + SQLAlchemy + JWT/RBAC。
- AI 与复用能力：`base_agent/` 和后端 `app/services/agent_service.py` 等服务层。
- 文档真源：`README.md`、`RUNNING.md`、`DEPLOYMENT.md`、`docs/` 和 `docs/adr/`。

除非用户明确要求并同步更新架构文档，否则不得把项目带向微服务、微前端、第二套前端框架、第二套权限体系或绕开现有 `/api` 边界的并行运行面。

核心业务语义必须有唯一 owner：

- 权限、组织隔离、角色边界和越权失败原因由后端 RBAC、中间件和服务层决定；前端只能消费和展示结果。
- BI 口径由后端统计逻辑、测试和 BI 文档共同约束；前端不得私自计算或改写业务真相。
- 招聘流程阶段、误推进修正、关闭/恢复、负责人转派、面试反馈归属等流程语义应收敛到后端服务或共享规则；页面按钮和弹窗只做触发与展示。
- AI 助手写操作必须继续走确认、RBAC、服务端校验和审计日志；不得用 prompt 文案、前端隐藏按钮或临时开关绕过。
- 页面、API、脚本和 worker 只是入口或适配层；跨入口复用的判断必须进入服务层、共享类型、测试或文档真源。

## 实现纪律

- 禁止用散落的 `if`、硬编码、mock、假数据、前端隐藏按钮、临时全局开关或固定文案绕过权限、BI、流程状态、AI 写操作和数据完整性问题。
- 修 bug 或加功能时，最小改动不等于半截改动；本轮触达的调用链必须闭合到对应 owner、测试和文档判断。
- 已有组件、API 客户端、服务、错误处理、审计和文档结构能满足需求时，优先沿用现有模式。
- 生成物、备份、上传文件、日志和历史迁移默认不手改；需要改动时先找到源头、脚本或明确的恢复流程。

## 验收分级

验收必须跟风险面匹配，不能只跑最轻命令就宣布完成：

- 后端权限、组织隔离、API 字段、流程状态和 AI 写操作：优先跑对应后端测试，并检查失败返回和审计事件。
- 前端页面、空态、按钮、权限可见性和用户流程：优先跑对应前端测试；必要时用真实页面检查桌面/移动、加载态、空态和错误态。
- BI 指标、统计归属和管理看板：同时检查后端统计测试、前端展示和 `docs/03_BI看板设计.md` 等口径文档。
- 端口、环境变量、部署、启动方式和数据清理：同步检查 `RUNNING.md`、`DEPLOYMENT.md`、脚本说明和实际命令结果。
- 如果关键验收无法运行，最终回复必须说明原因、影响范围和替代证据。

## Git 边界

- 开始较大改动前先看 `git status --short`，确认当前工作区是否已有用户或其他 agent 的改动。
- 禁止使用 `git add .`；只 stage 与本任务直接相关的文件。
- 不回滚、不覆盖、不格式化用户未授权的改动；如果用户改动影响当前任务，先说明冲突和推荐处理方式。
- 提交或交付前必须说明本轮 diff 范围、实际验证命令、未验证项和文档同步情况。

## 试点 / 上线判断规则

当用户询问“是否可以试点上线”“是否已经相对完善”“是否可以发给真实用户测试”时，不能只按主流程能跑通、自动化测试通过或核心权限正常来判断。必须先区分：

- **技术可运行**：服务能启动、接口能通、自动化测试通过。
- **产品可试用**：不同角色能完成主要工作，遇到缺数据、误操作、空列表时也知道下一步怎么办。
- **生产可上线**：具备真实数据安全、备份、合规、审计、监控和运维保障。

在给出“可试点”“可上线”“相对完善”等结论前，必须完成并说明以下审查：

- **角色视角审查**：分别从招聘专员、招聘经理/负责人、面试官、管理员视角走关键路径。
- **缺口入口审查**：检查岗位、候选人、面试官、账号、权限、BI 口径、AI 助手写操作等入口在“没有可选项、数据为空、数据被关闭/停用、用户无权限”时是否有清楚提示和下一步入口。
- **反向操作审查**：检查关闭后能否恢复、推进后能否撤回或修正、负责人能否调整、错误数据能否补救；暂不支持的必须作为风险说明。
- **数据归属审查**：检查候选人负责人、流程推进人、面试反馈人、岗位归属和 BI 统计口径是否一致，避免“看得到但算不清”。
- **文档和限制审查**：检查运行方式、账号密码、试点说明、已知限制和下一步优化是否写清楚。

如果以上审查没有完成，最终回复不得笼统承诺“可以上线”或“已经很完善”，只能说“技术主流程可运行”或“仍需补产品试点检查”。

## 用户视角验收规则

涉及页面、流程、权限、数据看板或试点体验的改动，除自动化测试外，还必须用普通用户语言检查：

- 用户第一眼是否知道该做什么。
- 用户没有数据时是否知道去哪里创建或导入。
- 用户选不到岗位、候选人、面试官、账号时是否有明确原因和下一步。
- 用户误关闭、误推进、填错信息时是否有补救路径；如果没有，必须在风险里说明。
- 管理者看到的数据是否能解释“谁做了什么、推进了多少、卡在哪里、责任在谁”。
